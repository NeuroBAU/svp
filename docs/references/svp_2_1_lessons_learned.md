# SVP 2.2 Lessons Learned — Bug Catalog

## Part 1: Bug Catalog

### Bug S3-1: state_hash spec ambiguity (Unit 5)

- **Bug number:** S3-1
- **How caught:** Stage 3 test generation — test agent wrote impossible self-referential hash assertion
- **Test file:** `tests/regressions/test_bug_s3_1_2_4_5_spec_clarity.py`
- **Description:** Spec Section 22.4 said state_hash is "SHA-256 hex digest of the pipeline_state.json file on disk, read immediately before each build log write." But save_state writes the hash INTO the file, creating self-referential ambiguity. The hash in the file is always the hash of the previous file state, not the current one.
- **Root cause pattern:** P7 (Specification Omission) — spec was ambiguous about hash timing, leading to impossible test expectations.
- **Prevention rule:** When specifying hash-based integrity fields, always clarify the temporal relationship: hash of state before or after the current write.

### Bug S3-2: Stub sentinel format inconsistency (Unit 6)

- **Bug number:** S3-2
- **How caught:** Stage 3 stub generation — agent produced `# SVP-GENERATED-STUB` instead of `__SVP_STUB__ = True`
- **Test file:** `tests/regressions/test_bug_s3_1_2_4_5_spec_clarity.py`
- **Description:** Stub generation agents inconsistently produce sentinel format. Spec requires `__SVP_STUB__ = True  # DO NOT DELIVER` (Python-assignable variable). One agent produced a comment-style sentinel. The compliance scanner relies on exact string matching.
- **Root cause pattern:** P7 (Specification Omission) — spec defined the sentinel format but did not mandate that prepare_task.py inject it verbatim into the stub generation prompt.
- **Prevention rule:** When a downstream consumer (compliance scanner) relies on exact string matching, the upstream producer (stub generator) must receive the exact string from a single source of truth (LANGUAGE_REGISTRY), not improvise.

### Bug S3-3: Test agent writes stub-detection tests (Unit 9)

- **Bug number:** S3-3
- **How caught:** Stage 3 test generation — test agent used `pytest.raises(NotImplementedError)` for 6 CLI behavioral tests
- **Test file:** `tests/regressions/test_bug_s3_3_6_9_test_agent_prohibitions.py`
- **Description:** Test agent used `pytest.raises(NotImplementedError)` for behavioral tests. These test that the stub is not implemented — they don't test actual behavior. They always fail against the real implementation.
- **Root cause pattern:** P14 (Agent Definition Gap) — the test agent definition lacked an explicit prohibition against testing stub behavior instead of contract behavior.
- **Prevention rule:** Agent definitions must explicitly prohibit anti-patterns that agents are likely to produce. Prohibitions are more effective than positive instructions for preventing known failure modes.

### Bug S3-4: Stub sentinel prompt standardization (systemic)

### Bug S3-5: state_hash timing in save_state (spec + blueprint)

### Bug S3-6: Test agent uses pytest.skip instead of failing (Unit 11)

- **Bug number:** S3-6
- **How caught:** Stage 3 red run — 51 tests SKIPPED instead of FAILED, producing misleading red run result
- **Test file:** `tests/regressions/test_bug_s3_3_6_9_test_agent_prohibitions.py`

### Bug S3-8: Unit 14 dispatch bypasses Unit 6 transition functions

- **Bug number:** S3-8
- **How caught:** Stage 3 code review — orchestrator observed dispatch using deepcopy + direct field assignment instead of transition functions
- **Test file:** `tests/regressions/test_bug_s3_8_dispatch_transitions.py`
- **Description:** dispatch_gate_response and dispatch_agent_status set 80+ state fields directly via deepcopy instead of calling Unit 6 transition functions. Blueprint contracts explicitly require transition function calls. Direct assignment bypasses validation preconditions.
- **Root cause pattern:** P2 (State Management Assumptions) — implementation agent chose to bypass transition functions "to avoid validation conflicts" without understanding that the validation IS the point.
- **Prevention rule:** State transition functions exist to enforce invariants. Implementation agents must use them even when direct assignment appears simpler. If transition preconditions are too strict, the fix is to relax the preconditions, not bypass the functions.

### Bug S3-10: Stub generator produces unimportable module-level constants

- **Bug number:** S3-10
- **How caught:** Pass 2 red run — ImportError on `from src.unit_1.stub import ARTIFACT_FILENAMES`
- **Test file:** `tests/regressions/test_bug_s3_10_stub_constant_defaults.py`
- **Description:** SVP 2.2's stub generator produces bare type annotations for module-level constants (`X: Dict[str, str]`) instead of annotated assignments with defaults (`X: Dict[str, str] = {}`). Bare annotations at module level don't create attributes, so `import` fails.
- **Root cause pattern:** P7 (Specification Omission) — spec and blueprint specified stub behavior for functions (raise NotImplementedError) and class attributes (= None) but not for module-level constants.
- **Prevention rule:** Blueprint must specify stub behavior for every AST node type the generator encounters: functions, classes, class attributes, AND module-level constants.

### Bug S3-13: Stub filename uses unit number prefix (Unit 10)

- **Bug number:** S3-13
- **How caught:** Stage 3 stub generation -- generated files named `unit_N_stub.py` but imports expected `stub.py`
- **Test file:** `tests/regressions/test_bug_s3_13_14_command_cycle.py`
- **Description:** The stub generator produced output files named `unit_{N}_stub{file_ext}` (e.g., `unit_5_stub.py`). The unit number is already encoded in the directory path (`src/unit_N/`), so the filename should be `stub{file_ext}`. The redundant unit number in the filename caused import mismatches because test agents and implementation agents expected `from src.unit_N.stub import ...`.
- **Root cause pattern:** P7 (Specification Omission) -- spec did not explicitly specify the output filename convention for generated stubs.
- **Prevention rule:** When a generated artifact lives in a unit-specific directory, the filename should not redundantly encode the unit number. Spec must specify the exact output filename.

### Bug S3-14: update_state_main lacks --command dispatch path (Unit 14)

- **Bug number:** S3-14
- **How caught:** Stage 3 action cycle -- run_command action blocks had no POST command, breaking the six-step cycle
- **Test file:** `tests/regressions/test_bug_s3_13_14_command_cycle.py`
- **Description:** `update_state_main` only accepted `--phase` (for agent dispatch) and `--gate-id` (for gate dispatch). It had no `--command` path for command status dispatch. As a result, `run_command` action blocks could not include a POST command that invoked `update_state.py --command <type>`. The six-step action cycle was incomplete for command actions -- the orchestrator ran the command but had no POST step to advance pipeline state.
- **Root cause pattern:** P1 (Incomplete Dependency Specification) -- the update_state_main CLI did not enumerate the command dispatch path that routing depends on.
- **Prevention rule:** Every action type that routing emits must have a corresponding dispatch path in update_state_main. The CLI interface must cover all three dispatch modes: agent (--phase), gate (--gate-id), and command (--command).

### Bug S3-15: Oracle session activation missing transition functions (Unit 6, Unit 14)

### Bug S3-16: Stage 4 test_execution dispatch missing gate presentations (Unit 14)

- **Bug number:** S3-16
- **How caught:** Stage 3 code review -- dispatch_command_status for Stage 4 test_execution just increments red_run_retries without setting sub_stage for gate presentation
- **Test file:** `tests/regressions/test_bug_s3_16_stage4_dispatch.py`
- **Description:** `dispatch_command_status` for Stage 4 `test_execution` treated TESTS_FAILED and TESTS_ERROR identically: both just called `increment_red_run_retries(state)` without setting `sub_stage` for gate presentation or re-invocation. The spec requires three-way differentiation: TESTS_FAILED must present Gate 4.1 (`gate_4_1_integration_failure`); TESTS_ERROR must re-invoke the integration test author (set `sub_stage=None`); retries exhausted must present Gate 4.2 (`gate_4_2_assembly_exhausted`). The routing function `_route_stage_4` also lacked branches for `sub_stage="gate_4_1"` and `sub_stage="gate_4_2"`, meaning even if dispatch set the sub_stage correctly, routing would fall through to the default (invoke integration_test_author) instead of presenting the gate.
- **Root cause pattern:** P1 (Incomplete Dependency Specification) -- Stage 4 dispatch didn't differentiate failure modes per spec. The blueprint originally specified TESTS_ERROR as "same dispatch as TESTS_FAILED" but the spec's universal four-state rule required re-invocation.
- **Prevention rule:** Every test execution dispatch must implement the full four-state differentiation (PASSED/FAILED/ERROR/EXHAUSTED) with distinct state transitions per the universal dispatch rule. Routing must have explicit branches for every sub_stage value that dispatch can produce.

### Bug S3-47: Stub generator must use TYPE_CHECKING guards for upstream imports (Unit 10)

## Part 2: Pattern Catalog

| Pattern | Name | Description |
|---------|------|-------------|
| P1 | Incomplete Dependency Specification | Blueprint contracts don't enumerate all dependencies, allowing agents to guess wrong |
| P2 | State Management Assumptions | Implementation bypasses state transition functions, losing validation |
| P7 | Specification Omission | Spec is ambiguous or silent on a behavior, leading to inconsistent implementations |
| P14 | Agent Definition Gap (NEW) | Agent definition lacks explicit prohibition of a known anti-pattern, causing agents to produce defective output |
| P15 | Procedural Check Without Structural Enforcement (NEW) | A required check exists as instructions but not as a runnable test, allowing silent omission |

**P14 is a new pattern** identified during SVP 2.2 Stage 3. It applies whenever an LLM agent consistently produces a specific defective pattern that could be prevented by an explicit prohibition in the agent's definition. P14 bugs are systemic — they recur across units because the agent definition is shared.

## Part 3: Post-Delivery Plugin Assembly Lessons (Bugs S3-51 through S3-54)

### Lesson: Plugin Manifest and Component Assembly Gap

- **Bugs:** S3-51 (missing .claude-plugin dirs), S3-52 (missing agents/commands/hooks/skills dirs), S3-53 (empty manifest fields), S3-54 (pass_transition routing)
- **Discovery:** After Pass 2 delivery, attempting to install SVP 2.2 as a Claude Code plugin failed completely. The delivered repo had only `svp/scripts/` but none of the required plugin infrastructure.

Validation (Gate C) should verify that all required directories exist and have content, not just that manifest schemas are valid.
- **Routing invariant:** Any function that calls `advance_sub_stage(state, X)` must also handle `sub_stage == X`, either in the same function or via a guaranteed dispatch path. The handler must not rely on falling through to a different stage's routing function.

- **Bug:** S3-55 (Pass 1 artifacts not synchronized to Pass 2 workspace)

- **Prevention pattern:** Any multi-pass build protocol that creates a new workspace for a subsequent pass must include an explicit sync step that merges accumulated non-code artifacts from the previous pass. - **Bug:** S3-65 (Oracle ~40% implemented -- routing skeleton but no operational logic)

### Lesson: Entry Point Script Completeness (Bug S3-66)

- **Bug:** S3-66 (routing.py and prepare_task.py missing __name__ guards)
- **Root cause:** Blueprint specifies function signatures but not script execution boilerplate. Implementation agents added __name__ guards inconsistently (2 of 4 scripts).
- **Prevention pattern P17:** Every script with main() must end with `if __name__ == "__main__": main()`. Every script using bare imports must add its directory to sys.path. These are NOT implementation details — they are execution requirements that must be in the blueprint.
- **Sweep extension (Bugs S3-130, S3-131, S3-132, S3-133):** The original S3-66 fix was scoped to the two scripts that had visibly failed. A post-delivery audit surfaced four additional scripts with the same defect — S3-130 `infrastructure_setup.py` (Unit 11), S3-131 `stub_generator.py` (Unit 10), S3-132 `quality_gate.py` (Unit 15), S3-133 `structural_check.py` (Unit 28). Each was invoked by routing.py as a subprocess but silently exited 0 without calling its main(). Lesson reinforcing P30 (Invariant Documented But Not Enforced): when a pattern is codified for the first time (P17 in 2.2), the initial fix must include an exhaustive sweep of all known instances of the pattern, not only the ones that manifested as bugs. Detection: `tests/test_entry_point_completeness.py` AST-level regression test added to prevent future instances.

### Lesson: Platform-Specific Stdlib Import Audit (Bug S3-134)

- **Bug:** S3-134 (ledger_manager.py unconditionally imports fcntl at module top-level; fcntl is Unix-only so Windows imports crash with ModuleNotFoundError)
- **Root cause:** Unit 7's blueprint specified JSONL concurrent-append semantics but did not constrain the locking primitive. The implementation agent chose fcntl.flock (natural on Unix) without a platform guard. The spec declares Python as primary language but does not enumerate supported host OSes — Windows parity was implicit and unenforced.
- **Prevention pattern P33 (NEW — Cross-Platform Stdlib Audit):** Every stdlib import must be checked for platform availability. Unix-only stdlib: fcntl, termios, pwd, grp, resource, select.kqueue/select.epoll. Windows-only: msvcrt, winreg, _winapi. When an operation requires platform-specific stdlib (file locking, terminal control, user DB, etc.), isolate it in a thin wrapper gated by `if sys.platform == "win32":` with both branches defined at import time. Do not use lazy `try/except ImportError` fallbacks — both paths must be parseable and loadable from every platform.
- **Detection:** `tests/test_ledger_cross_platform.py` — AST-structural test asserting that fcntl is never imported at top level in src/unit_7/stub.py, plus a functional test that `append_entry` succeeds on the current host. Full Windows validation requires a Windows CI runner (deferred).

### Lesson: Stage-Boundary Artifact Materialization (Bug S3-135)

- **Bug:** S3-135 (toolchain.json expected at project root but never materialized; Stage 3 entry crashes with FileNotFoundError)
- **Root cause:** The two-layer toolchain design (project-root pipeline toolchain vs. language-default template in scripts/toolchain_defaults/) is encoded in load_toolchain's signature, but the Layer 2 → Layer 1 promotion path was never written in code. Spec Section 8 describes toolchains conceptually without specifying where the materialization happens. Result: no unit owned the copy, so no unit did it.
- **Prevention pattern P34 (NEW — Stage-Boundary Artifact Materialization):** Every stage-output consumed as a file by a later stage must have an explicit materialization step owned by a single unit. The owner is either (a) the stage's approval-gate handler, or (b) the first consumer, self-healing via a helper. When a two-layer design exists (template vs. instance), the promotion path between layers must be encoded in code, not prose. Always add a regression test that exercises the materialization in both directions: "file exists → no-op" and "file absent → materialize from template + correct contents".
- **Detection:** `tests/unit_11/test_ensure_pipeline_toolchain.py` — 4 parametrized cases covering the no-op, python, R, and missing-profile paths.
- **Second consumer closed (Bug S3-136).** `_route_pre_stage_3` in Unit 14 was the routing-called counterpart to Unit 11's `main()`. Prior to the fix, the `if state.total_units > 0` branch was dead code because nothing ever populated `total_units`. The fix invokes `ensure_pipeline_toolchain` + `run_infrastructure_setup` as direct function calls (no subprocess), then reloads state. Takeaway: when applying P34, inventory every caller path that consumes the materialized artifact, not only the most obvious one.
- **Build-but-don't-execute variant closed (Bug S3-137).** `run_infrastructure_setup` Step 1 built the conda env-creation command into `env_info["commands"]` and then never executed it; the inline comment said "execution is deferred to the orchestration layer" but no caller picked up the work. The fix adds a new `_env_exists` helper and a Step 4b execution block that actually invokes `subprocess.run` on the built commands and the consolidated install list (quality tools + bridge packages + third-party runtime deps). Takeaway: a comment saying "the caller will run this" is a design smell — either name the caller explicitly or execute inline.

### Lesson: Producer-Consumer Contract Consistency (Bug S3-138)

- **Bug:** S3-138 (`run_quality_gate_main` printed QUALITY_* to stdout and exited 0 always; orchestrator translated every exit-0 to `COMMAND_SUCCEEDED` per spec §3.7, silently advancing past failing gates).
- **Root cause:** Two sides of the producer-consumer contract were designed independently. The producer (quality_gate.py) thought stdout carried the status signal; the consumer (routing.py::dispatch_command_status) expected `COMMAND_*` strings derived from exit code per spec §3.7. Neither side was wrong in isolation, but they disagreed on the channel, and the spec arbitrates in favor of exit code.
- **Prevention:** For every CLI tool invoked as a `run_command` by the orchestrator, assert two invariants: (1) the tool's exit code is authoritative for success/failure (exit 0 == success, nonzero == failure), (2) stdout is diagnostic only. Document the exit-code contract in the tool's entry-point docstring, not just in the spec.
- **Detection:** `tests/unit_15/test_s3_138_quality_gate_exit_codes.py` — parametrized exhaustive coverage of the 4 QUALITY_* statuses mapping to 2 exit codes.
- **Consumer-side aligned (Bug S3-139).** Spec §18.3 defines the failure status as `COMMAND_FAILED: [exit code]` (with the code appended). All four quality_gate branches of dispatch_command_status used strict `==` against bare `"COMMAND_FAILED"`, which missed the spec-defined suffix form and crashed routing with ValueError. Fixed by switching to substring matching (`"FAILED" in status_line`), consistent with compliance_scan / structural_check / lessons_learned branches.

### Lesson: Schema-Consumer Wiring Completeness (Bug S3-140)

- **Bug:** S3-140 (toolchain schema declared `unit_flags` / `project_flags` first-class fields with concrete values like `"--ignore-missing-imports"` for mypy, but `_execute_gate_operations` never read them; `resolve_command`'s `flags=""` parameter was never populated. Mypy ran without the flag at every Stage 3 unit gate, producing false-positive import-not-found residuals on every test file.)
- **Root cause:** Schema-declared fields with no consumer. `resolve_command`'s `flags=""` default is a hint that propagation was once intended; the wiring was never completed. The toolchain JSON and the gate-runner agreed on the shape of `unit_flags`/`project_flags` but not on who would read them.
- **Prevention:** When a configuration schema declares first-class fields, every consumer must either read them or explicitly document why they are ignored. A silently-ignored field is a latent bug; `resolve_command`'s accepting-but-not-requiring `flags=` parameter absorbed the defect for an unknown span of time. Default-empty placeholder substitution is a sharp corner — if a template has `{flags}` and the consumer passes nothing, the resulting command silently differs from the schema's intent.
- **Detection:** `tests/unit_15/test_s3_140_unit_flags_propagation.py` — asserts `resolve_command` receives the expected `flags=` kwarg for each tool's `unit_flags` configuration.

### Lesson: Routing-Agent Mode Handshake (Bug S3-141)

- **Bug:** S3-141 (`_agent_prepare_cmd` in Unit 14 never forwarded `--mode` to prepare_task.py; the three setup_agent call sites in `_route_stage_0` emitted identical PREPARE commands despite their reminders differing by sub_stage. Setup_agent in `project_context` sub_stage defaulted to `project_profile` behavior, producing `PROFILE_COMPLETE` instead of `PROJECT_CONTEXT_COMPLETE`, and routing looped without progress.)
- **Root cause:** The generic `_agent_prepare_cmd` builder had no mode parameter. Over time, specific `_prepare_<agent>` functions in Unit 13 grew mode-dependent branching (setup_agent, stakeholder_dialog, blueprint_author), but the routing builder was never extended. The reminder string in affected action blocks carried the correct sub_stage context, indicating routing KNEW the mode — it just didn't thread it into the command.
- **Prevention:** When an agent's `_prepare_<agent>` accepts mode-dependent parameters, every `_agent_prepare_cmd` call site must pass `--mode`. A minimal AST-level regression test: for every agent type invoked from multiple sub_stages, verify each invocation passes a mode kwarg to `_agent_prepare_cmd`. Alternatively: whenever two action blocks share an `agent_type` but differ by sub_stage (detected by reminder diff or by routing branch location), the PREPARE commands must also differ.
- **Detection:** `tests/unit_14/test_s3_141_setup_agent_mode.py` — asserts both signature behavior of `_agent_prepare_cmd(mode=...)` and routing-level behavior at each Stage 0 sub_stage.
- **Applied to stakeholder_dialog (Bug S3-142).** Two of the three stakeholder_dialog call sites in routing (`_route_stage_1` + `_route_stage_2`, both at `sub == "targeted_spec_revision"`) now pass `mode="targeted_revision"`. The third site (Stage 1 main fall-through) is deferred because it conflates initial-dialog and post-REVISE rerun with no state-level distinguisher — a future cycle will add one before attempting mode threading there. Lesson: when applying a routing-agent mode handshake, scope the fix to unambiguous sites and explicitly defer ambiguous ones rather than guessing.
- **Deferred in full for blueprint_author (Bug S3-143).** Both blueprint_author call sites in `_route_stage_2` share the same initial-vs-revise ambiguity as the Stage 1 fall-through (§24.155), AND `_prepare_blueprint_author`'s "Reviewer Feedback" section requires BOTH `mode == "revision"` and non-empty `context` — neither of which routing threads. Threading `mode` alone would be cosmetic (adds only a "Mode: revision" header; no new content loads). Shipping a cosmetic change would mislead future readers into thinking the work is done. Lesson: **scope-honesty matters**. When the minimum-change scope reduces to cosmetic-only, ship no change and document the deferral rather than a placebo fix. Anchor tests pin current behavior so a future cycle can break them deliberately when the full signal chain lands.

### Lesson: File-vs-Stdout Should Be Mutually Exclusive (Bug S3-144)

- **Bug:** S3-144 (`prepare_task.py::main` wrote to `--output` AND echoed to stdout unconditionally; on Windows, the stdout print crashed with UnicodeEncodeError for task prompts containing non-cp1252 characters — box-drawing glyphs, em-dashes, typographic ellipsis. The file write succeeded before the crash, but exit 1 + traceback polluted every Stage 1+ PREPARE.)
- **Root cause:** Redundant dual-output. The stdout print served no consumer — the orchestrator only reads the output file, never stdout. The duplication was the sole source of the cp1252 crash: file write used `encoding="utf-8"`, while stdout relied on `sys.stdout.encoding`.
- **Prevention:** When a script accepts both a file-output flag and produces stdout output, prefer mutual exclusion (if `--output` set, write file and return silently; else print to stdout). A dual channel is only justified when a specific consumer reads both — otherwise it is dead weight that introduces platform-encoding bugs at zero functional benefit.
- **Detection:** `tests/unit_13/test_s3_144_output_vs_stdout.py` — exercises both branches with `capsys` to verify the mutual-exclusion contract.

### Lesson: Test Fixtures Must Model Reality, Not the Code Under Test (Bug S3-145)

- **Bug:** S3-145 (`svp_launcher.create_new_project` referenced `plugin_root/"svp"/"hooks"` for the hook-deployment source, but the real plugin cache lays hooks at `plugin_root/"hooks"`. The deployment loop silently failed because the directory didn't exist, `.claude/scripts/` was never populated, and `non_svp_protection.sh` failed open. The existing regression test's fixture built hooks at the same wrong path, so the test passed while validating the wrong reality — a mirror-the-bug fixture.)
- **Root cause:** Pattern P30 (Invariant Documented But Not Enforced) manifested in the test layer. The hook-deployment intent was recorded in the S3-108 fix comment, but the path assumption was never verified against the real plugin cache layout. The regression test inherited the assumption.
- **Prevention:** Test fixtures must model real deployment layouts, not the layouts assumed by the code under test. When fixing a path or layout bug, audit adjacent test fixtures for the same mirror-the-bug pattern. A test that passes AND validates against a real-world sample (e.g., an assertion that a constant matches `~/.claude/plugins/cache/...` structure) is stronger than a self-consistent synthetic fixture.
- **Detection:** `tests/regressions/test_bug_s3_145_plugin_hooks_path.py` pins the corrected fixture layout against future drift.

### Lesson: Agent-Discretionary Step Promoted to Code Path (Bug S3-146)

- **Bug:** S3-146 (Stage 5 Python project assembly did not deliver `tests/`. Spec assumed the agent would copy `workspace/tests/` to `repo/tests/` and adapt flat-import test files to match the delivered source layout, but the agent prompt never explicitly required the copy and the assembler had no helper for either step. On Windows-advancing pipelines the agent's ad-hoc reasoning produced an empty `tests/` in the delivered repo.)
- **Root cause:** Pattern P30 (Invariant Documented But Not Enforced) at a workspace-to-repo boundary. The step's existence was specced in prose (§12.1.X line 2962) but no code path implemented it; the agent was the implicit implementation, which fails non-deterministically.
- **Prevention pattern P35 (NEW — Agent-Discretionary Step Promoted to Code Path):** When a workspace-to-repo delivery step has been specced informally as "the agent does X", and X has both a deterministic implementation AND a track record of agent-discretion failures, promote X to a code helper invoked from the assembler. Update the agent prompt to forbid re-doing X manually. P35 covers a broader pattern than P34 (Stage-Boundary Artifact Materialization) because the trigger is "informally-specced agent step", not necessarily a multi-stage handoff. The routing-mode handshake cycles (S3-141, S3-142, S3-143) are earlier instances of this pattern; future audits should look for "the agent does X" prose without a corresponding helper as a candidate.

### Lesson A: String Assertion Tests on Hand-Authored Fixtures Do Not Validate Generators (Bug S3-109)

- **Bug:** S3-109. `_write_pyproject_toml()` in Unit 23 hardcoded `build-backend = "setuptools.backends._legacy:_Backend"` — a fictional module. Every delivered Python project's `pyproject.toml` was unbuildable: `pip install -e .` failed immediately with "Backend not available". The correct PEP 517 value is `setuptools.build_meta`. An existing regression test `TestPyprojectToml.test_build_backend` asserted `"setuptools.build_meta" in content` and passed — yet the generator was emitting the fictional string. The test read `svp2.2-pass2-repo/pyproject.toml`, which is a HAND-AUTHORED file that was already correct, instead of invoking `assemble_python_project()` on a tmp dir and parsing its OUTPUT. The test gave false comfort by validating the wrong artifact.
- **Root cause:** Pattern P-NEW — "Fixture-tests-fixture": a regression test that asserts on a string in a hand-authored file provides ZERO assurance about the generator that would produce a similar-looking file at runtime. The test author assumed "the string is in the committed file → the generator must produce it," but the committed file and the generator are independent artifacts maintained by different mechanisms. When the generator string was wrong, the hand-authored file remained right, and the test remained green. This is structurally indistinguishable from not having a test at all.
- **Prevention:** (1) Any regression test that asserts on text emitted by a generator MUST invoke the generator and parse its output. Never read a committed fixture as a proxy. (2) When adding a new regression test for "artifact X must contain Y", first ask: "What code produces X at delivery time?" Test THAT code. (3) For any generator of a well-known format (pyproject.toml, JSON schemas, YAML configs), use the format's native parser (`tomllib`, `json`, `yaml`) on the generator's output — string-`in` checks are weaker than structural assertions and easier to spoof by coincidence. (4) When the generated artifact must satisfy an external contract (e.g., PEP 517 build backend must be importable), add a runtime verification step: `subprocess.run([sys.executable, "-c", "import <backend>"])` — this is the only way to catch "plausible-looking but fictional" strings.

### Lesson B: Incomplete Refactors Leave Orphaned Duplicate Source Files (Bug S3-109, related to S3-98)

- **Discovered while fixing:** S3-109 (the build-backend bug was duplicated in two places because Unit 23 had been incompletely refactored).
- **The situation:** Pre-S3-98, Unit 23 was a "composite unit" whose source lived in multiple sibling files under `src/unit_23/` (one per deliverable: `adapt_regression_tests.py`, `generate_assembly_map.py`, `git_repo_agent.md`, `checklist_generation.md`, etc.). S3-98's "derive scripts from stubs" refactor collapsed all of these into `src/unit_23/stub.py` and introduced `scripts/derive_scripts_from_stubs.py` with a `STUB_TO_SCRIPT` dict. But that dict only maps ONE output per stub, and for Unit 23 it mapped to `scripts/generate_assembly_map.py`. The second historical output — `scripts/adapt_regression_tests.py` — was never added to the map, never re-derived, and was never deleted. It sat in `scripts/` as a hand-maintained stale copy containing pre-refactor code (missing 5 newer assembly functions). `src/unit_11/stub.py:703` still invokes it by path during infrastructure setup step 8. The duplicated `_write_pyproject_toml()` in the orphan was unreachable via Unit 11's current `--target/--map` call shape, but it was a latent time bomb: any future caller that uses `assemble_python_project` from the orphan would hit the same bug.
- **Root cause:** P-NEW — "Multi-output unit collapsed to single-output derivation without deleting or re-wiring orphans". The S3-98 refactor transformed the Unit 23 data flow from "N source files in src/unit_23/ → N derived scripts" to "1 stub.py → 1 derived script", but did not audit which scripts existed in `scripts/` that no longer had a source of truth. The `.svp/assembly_map.json` still lists the pre-refactor per-file source paths (`src/unit_23/adapt_regression_tests.py`, etc.) that no longer exist on disk, which is a related stale-configuration bug.
- **Prevention:** (1) Whenever a refactor COLLAPSES multiple source files into a single source of truth, the refactor MUST either (a) add all historical outputs to the new derivation map so every downstream consumer gets re-generated content, or (b) delete each orphaned file and update every caller to use the new name. Partial migrations are forbidden — either all outputs are live, or all orphans are deleted. (2) After any refactor that changes the `STUB_TO_SCRIPT` map or the assembly map, run a consistency check: every file in `scripts/` must either (i) be in `STUB_TO_SCRIPT` as a derivation target, or (ii) be an entry-point CLI with no stub counterpart. Neither-nor is an orphan and must be flagged. (3) `.svp/assembly_map.json` must be validated against the filesystem after every structural refactor: for every source path in the map, assert `Path(src).exists()`. Stale map entries are silent correctness bugs.
- **Follow-up bugs filed (not fixed in S3-109):** (1) Delete or properly derive `scripts/adapt_regression_tests.py` (two options: extend `STUB_TO_SCRIPT` to multi-output, or delete and redirect Unit 11 to `generate_assembly_map.py`). **Fixed as Bug S3-110 — see next entry.** (2) Audit `.svp/assembly_map.json` for stale per-unit source paths (multiple `src/unit_23/*.md` entries point at files that no longer exist). **Still open.**

### Lesson: Deletion Over Wrapping for Internal Callers (Bug S3-110)

- **Bug:** S3-110 (follow-up to S3-109). `scripts/adapt_regression_tests.py` was an orphaned 881-line duplicate of Unit 23 code surviving from pre-S3-98 composite-unit layout. The only caller was `src/unit_11/stub.py:703` (internal, editable). Two fix options were considered: (a) thin wrapper — rewrite `adapt_regression_tests.py` as ~20 lines that import and dispatch, or (b) delete entirely — add a CLI to the derived script `generate_assembly_map.py` via a new `__main__` block in the stub, redirect Unit 11 to the new entry point. Option (b) was chosen and implemented.
- **Root cause:** Incomplete S3-98 migration (composite unit → single stub) that wired only ONE derived output into `STUB_TO_SCRIPT`, leaving the second historical output as an orphan.
- **Why deletion over wrapping:** A wrapper introduces an indirection layer (`adapt_regression_tests.py` imports from `generate_assembly_map.py`) that preserves the old filename but adds no value. The filename was not a stable external API — the only caller was internal code we owned and could edit. Deletion removes the filename entirely, updates the caller, and leaves a cleaner final architecture: one source of truth (stub), one derived script (with CLI subcommands), zero orphans, zero wrappers. The diff is larger (multiple files touched) but the end state is structurally simpler.
- **Prevention:** When consolidating multiple historical filenames into a single source of truth: (1) Audit every caller of the orphan. If ALL callers are internal (editable in the same codebase), prefer deletion + redirect over wrapper. (2) A wrapper is acceptable only when an orphan filename is a stable external contract — e.g., it's documented in a user-facing manual, or it's referenced by a tool you don't control. (3) When redirecting to a CLI subcommand, verify the flag names match: the new dispatcher's `--target/--map` did NOT match the inner function's `--tests-dir/--map-file`, which caused a smoke-test failure caught during S3-110 execution. Always run the subprocess end-to-end against a tmp directory with known input/expected output — help text alone is insufficient. (4) Remove stale entries from `.svp/assembly_map.json` in the same commit as the deletion; orphaned map entries are silent correctness bugs (see FOLLOW-UP for full map audit).

### Lesson: Generators Must Co-Migrate With Layout Changes (Bug S3-111)

- **Bug:** S3-111 (second follow-up to S3-109 / S3-110). During the S3-110 cleanup, an audit of `.svp/assembly_map.json` revealed that **every one of its 70 forward-direction entries pointed at source files that do not exist**. The function `generate_assembly_map()` in `src/unit_23/stub.py` at line 1221 constructed workspace paths using a hardcoded pre-S3-98 formula `f"src/unit_{N}/{name}"` — but post-S3-98, every unit's source is a single `src/unit_N/stub.py` file. The generator was never updated when S3-98 collapsed the composite-unit layout. Because the function is called during every Stage 5 reassembly, it deterministically regenerated 100% stale data on every run. The S3-110 hand-fix (removing two `adapt_regression_tests.py` entries from the map) would have been silently undone on the next sync.
- **Root cause:** The function encoded the old file layout as a string formula. When the layout changed, the formula did not. Unlike the "config drifts from code" pattern (S3-104/105/106/107, where `ARTIFACT_FILENAMES` declared one path and code hardcoded another), here the drift is between **code structure** and a **generator that produces data about code structure**. The bug was silent because the forward direction had no live consumers: Unit 13 embeds the map as raw text in agent task prompts (no JSON parsing), Unit 24's debug agent uses only the reverse direction, Unit 23's git repo agent text mentions the map but the actual assembly uses `regenerate_deployed_artifacts()` and `derive_scripts_from_stubs.py` instead of iterating the map. Tests checked bijectivity but never file existence.
- **Fix:** Rewrote `generate_assembly_map()` to produce a flat `{"repo_to_workspace": {...}}` schema. Dropped `workspace_to_repo` entirely — the forward direction was neither needed nor semantically representable (many-to-one can't fit in `Dict[str, str]`). Changed the source path formula from `f"src/unit_{N}/{name}"` to `f"src/unit_{N}/stub.py"`. Added regression tests asserting every value matches `^src/unit_\d+/stub\.py$` AND points at a file that exists on disk — the staleness check that would have caught the original bug.
- **Pattern:** P4 (Config-Code Divergence, extended). Fifth instance of the pattern in SVP 2.2. Previous instances were hardcoded artifact paths; this one is a hardcoded path **formula** inside a generator. Same root cause: something that should derive from a canonical source encodes the canonical source as a constant, and the constant drifts.
- **Prevention:**
    1. **Audit generators during layout changes.** When you refactor file layout (composite unit → single stub, flat → hierarchical, etc.), grep the entire codebase for string formulas that encode the old layout. Any `f"src/unit_{N}/..."`, `f".../scripts/{module}"`, or similar is a candidate for co-migration. Update them in the same commit as the layout change.
    2. **Test file existence, not just structure.** A bijectivity test that checks `A ↔ B` holds is useless if neither A nor B actually exists on disk. Always include `assert Path(value).exists()` in regression tests for any generator that produces path data.
    3. **Detect silent drift.** A generator that silently produces stale data is worse than one that fails loudly. If you find yourself hand-fixing generator output (as in S3-110), that's a signal the generator itself is wrong — don't patch the data, patch the function.
    4. **Question the map's direction needs.** Before migrating a two-direction dict, ask: does any live code consume the forward direction? Often only the reverse is needed. Dropping unused directions removes a class of bugs.
    5. **Update agent-facing documentation.** Generator text constants embedded in agent definitions (like `GIT_REPO_AGENT_DEFINITION`) propagate schema drift to the LLM consumers. When changing a schema, update the constant in the same commit; the sync pipeline (`regenerate_deployed_artifacts()`) will then rewrite the deployed `.md`.

### Lesson: Agent Discretion on Deterministic Facts Is a Liability (Bug S3-112)

- **Bug:** S3-112 (reported externally on 2026-04-12 against SVP 2.2.0). The `git_repo_agent` chose `./delivered/` as its destination directory instead of the canonical sibling `../{project_name}-repo/`. `state.delivered_repo_path` was written as a bare relative string `"delivered"` instead of an absolute path. Every downstream consumer (sync_debug_docs, bug_triage_agent path correlation, write-authorization hook's `*-repo/*` allowlist, Oracle nested session) silently broke. Stage 5 still reported `pipeline_complete` — the misplacement did not surface as a gate failure.
- **Root cause:** Agent discretion on a deterministic fact. The delivered repo path is fully computable from `(project_root, project_name)` — it is `project_root.parent / f"{project_name}-repo"` by convention, and this rule was encoded only inside `assemble_python_project()` / `assemble_r_project()` helper implementations. The pipeline code: (a) did not compute the path in the prepare step, (b) did not inject it into the agent's task prompt as a constraint, (c) did not compute or validate it in the POST dispatch step, (d) did not name the required helper functions for the agent to call. `dispatch_agent_status` for `git_repo_agent + REPO_ASSEMBLY_COMPLETE` was a literal no-op `_copy(state)`. With zero binding from any direction, the LLM was free to improvise — and did. In pass2's own build, the agent happened to do it right, but that was luck. In `debrief1.0`, it picked `./delivered/`.
- **Fix:** Defense-in-depth three-layer enforcement.
    1. **Task prompt injection** (`_prepare_git_repo_agent` in Unit 13): compute `canonical_delivered_path = (project_root.parent / f"{profile_name}-repo").resolve()` and inject it as a "Delivered Repo Path (REQUIRED)" section with a directive to call the named assembler helper.
    2. **Explicit agent binding** (`GIT_REPO_AGENT_DEFINITION` in Unit 23): new `## Delivered Repo Location` section stating the sibling convention, naming all four `assemble_*_project()` helpers, forbidding `delivered/`/`delivered_repo/`/`output/` and manual state edits.
    3. **POST dispatch validation** (`dispatch_agent_status` for `git_repo_agent + REPO_ASSEMBLY_COMPLETE` in Unit 14): load profile, compute canonical path independently, verify directory exists, call `set_delivered_repo_path(state, str(canonical_path.resolve()))`. Raise `ValueError` with a clear Bug S3-112 reference if the directory does not exist.

  Layers 1 and 2 nudge the agent to do the right thing; Layer 3 is the hard guarantee that `state.delivered_repo_path` is always absolute, always sibling, always exists on disk regardless of what the agent did.

- **Pattern:** P4 extension, sub-pattern **"Agent Discretion on Deterministic Facts Is a Liability."** Sibling of S3-104/105/106/107/111 in the Config-Code Divergence family. Previous instances were hardcoded paths diverging from canonical config; this instance is an **absence** of canonical code entirely, leaving the LLM free to improvise. The fix template is the same each time: identify the fact, compute it in deterministic code, inject/validate in both directions (task prompt → agent, POST dispatch ← agent).
- **Prevention rules:**
    1. **Rule of thumb**: any fact that is deterministically computable from pipeline inputs — paths, names, hashes, derived schemas, canonical destinations — MUST be computed by pipeline code and NEVER by the LLM. If the LLM can see the fact, inject it as a REQUIRED task-prompt section. If the pipeline will consume the fact downstream, compute AND validate it in the POST dispatch step.
    2. **"No-op dispatch is a red flag."** Any `dispatch_agent_status` branch that returns `_copy(state)` is a place where the agent has free rein over something. Audit each one: what facts does the agent implicitly set? Are any of them deterministic? If yes, replace the no-op with compute-validate-set.
    3. **"When state holds a path, it must be absolute."** Any state field documenting a filesystem location should be typed/validated as absolute in the POST dispatch step. A relative string in state is almost always an agent-discretion failure — grep for it and add the validator.
    4. **Task prompt injection + agent binding + POST validation = defense in depth.** Each alone is leaky. Task prompt alone: agent can ignore. Agent binding alone: documentation can be misinterpreted. POST validation alone: fires after the damage. Together they eliminate the failure mode.
    5. **When a bug report flags a "silent" failure at the plugin boundary** (the pipeline reports success, but downstream tooling breaks), the root cause is almost always a missing invariant check at the handoff. Add the check where the handoff happens (POST dispatch), not where the downstream breaks.

### Lesson: Content Validation at the Delivery Handoff (Bug S3-113)

- **Bug:** S3-113 (follow-up to S3-112). S3-112 closed the agent-discretion gap at the PATH level, guaranteeing that `state.delivered_repo_path` is absolute, sibling, and exists on disk after `REPO_ASSEMBLY_COMPLETE`. But existence ≠ correctness. An empty directory passes. A directory with stale files passes. A Python project missing `pyproject.toml` passes. A project whose delivered files do not match the assembly map passes. The pipeline reports `pipeline_complete` and the human believes delivery succeeded, but downstream tooling breaks on content issues that only surface much later.
- **Interesting self-reference**: the original S3-112 bug report's suggestion 5 was to add a `compliance_scan` check for path correctness. I rejected it as redundant with S3-112's Layer 3 (both check existence). I was right for the wrong reason — existence WAS already covered, but **content** was not. S3-113 re-adopts suggestion 5 for a different purpose: content validation, not existence. The reporter's instinct was correct; my rejection was too narrow.
- **Fix:** Add `validate_delivered_repo_contents(project_root) -> List[Dict[str, Any]]` in Unit 28, wired into `compliance_scan_main` to merge its findings with the existing language-specific compliance scanners. Three checks:
  1. Required root-level delivery files present (language-dependent list: `pyproject.toml`+common for Python; `DESCRIPTION`+`NAMESPACE`+common for R).
  2. Assembly-map parity: every key in `.svp/assembly_map.json`'s `repo_to_workspace` resolves to an existing file inside `delivered_repo_path` (with the `svp-repo/` prefix stripped).
  3. Python `pyproject.toml` validity: parses as TOML, has `[build-system]`, `build-backend == "setuptools.build_meta"`. This directly prevents regression of Bug S3-109 in the delivered artifact (the existing S3-109 test checks the generator's tmp-dir output; this check validates what was actually written).
- **What the new check caught immediately**: on first run against the current workspace, `validate_delivered_repo_contents` flagged that `svp-repo/tests/regressions/test_profile_migration.py` was declared in the assembly map but did not exist on disk. The blueprint's Preamble file tree annotated it as `<- Unit 3 (NEW IN 2.2)` and the spec Section 1536 described it — but nobody had ever written the file. It was a long-standing drift. As part of the S3-113 fix, I created `tests/regressions/test_profile_migration.py` with 9 real tests covering the SVP 2.1 → 2.2 profile migration logic in `load_profile()`. The S3-113 content check paid for itself on the first run.
- **Patterns:**
  - **"Validate at the handoff boundary, not at the consumer."** The delivered repo is the handoff between SVP's pipeline and the user's install/use. Check invariants there, not in `sync_debug_docs.py` or the oracle agent (which are the consumers). `compliance_scan` is the correct home because it already runs as the last Stage 5 gate before `pipeline_complete`.
  - **"Existence checks and content checks are different layers."** Rejecting content validation as "redundant with existence validation" (as I did initially with S3-112 reporter's suggestion 5) conflates the two. Always ask: is this check about the shell or the filling?
  - **"A content check will find drift you didn't know about."** Adding `validate_delivered_repo_contents` immediately surfaced the missing `test_profile_migration.py`. When you add a new invariant check, the first run is a free audit — expect to find at least one pre-existing violation. Treat each finding seriously; do not suppress them.
- **Prevention:**
  1. When adding a new invariant check, always run it against the current-state of the repo FIRST, before any test-time assertions. Existing drift will surface as findings; fix those drifts as part of the same bug fix (scope creep is acceptable because the check paid for itself).
  2. Distinguish "check the shell" (existence, path correctness, reachability) from "check the filling" (contents match expectations). Do not conflate.
  3. For generator-produced artifacts, test BOTH the generator output (unit test on a tmp dir) AND the delivered artifact (regression test or compliance scan). The S3-109 test covered only the generator; without S3-113's content check, a stale committed pyproject.toml could silently diverge from what the generator now produces.

### Lesson: Routing Recursion Must Always Advance State (Bug S3-114)

- **Bug:** S3-114 (reported externally). During Stage 2 of a real SVP project, the blueprint checker emitted `ALIGNMENT_FAILED: blueprint`. The operator attempted a workaround by writing the status directly to `.svp/last_status.txt` and re-invoking `routing.py` (bypassing `update_state.py`). Routing hit infinite recursion at `_route_stage_2` and crashed with a stack overflow. The branch at `src/unit_14/stub.py:1547` called `return route(project_root)` without first advancing state on disk; the recursive call re-read the same state, entered the same branch, and recursed again — forever.
- **Root cause:** The `alignment_check + ALIGNMENT_FAILED + iterations<limit` branch skipped the `advance_sub_stage → save_state` step that the parallel `ALIGNMENT_CONFIRMED` branch (line 1531-1534) correctly performs. The regression was structural, not behavioral: two branches that should follow the same recursion-before-advance pattern diverged.
- **When does the broken branch fire?** When dispatch was skipped. Dispatch's successful path (`dispatch_agent_status` for `blueprint_checker + ALIGNMENT_FAILED: *` at lines 2500-2511) advances sub_stage to `"blueprint_dialog"` or `"targeted_spec_revision"`, so the next `route()` call hits a DIFFERENT branch. The broken branch is reachable only when sub_stage is STILL `alignment_check` at route time — which happens when dispatch didn't run.
- **Fix (Bug S3-114):** Make routing self-heal for this case. Mirror dispatch's state transition inline: increment iterations, advance sub_stage (to `"blueprint_dialog"` for blueprint failure, `"targeted_spec_revision"` for spec failure), save_state, THEN recurse. Guaranteed safe because the post-advance state takes routing to a different branch. No double-count because dispatch-ran paths never enter this branch.
- **Pattern:** **"If you `return route(project_root)`, the state on disk MUST differ from what the current `route()` call read."** Otherwise you recurse forever through the same branch. This is a structural invariant for every recursive-routing call site in SVP.
- **Prevention rules:**
  1. **Grep `routing.py` for `return route(project_root)`** and audit each call site. Every one MUST be preceded by a state-advancing operation (`advance_sub_stage`, `complete_*`, `increment_*`, `restart_from_stage`, etc.) AND a `save_state(project_root, state)` call. No exceptions.
  2. **Add a bounded-recursion regression test** for every recursive routing branch. Use `sys.setrecursionlimit(100)` (deterministic, fast) rather than timeouts (flaky). The test asserts that `route()` returns within the recursion budget, catching infinite loops immediately.
  3. **Routing is the "last line of defense" against skipped dispatch.** Even though the canonical flow is `dispatch → routing`, routing should self-heal (mirror dispatch's state transitions) rather than rely on dispatch having been called correctly. Operators can and will bypass dispatch in emergencies; routing must degrade gracefully instead of hanging the process.
  4. **When you find a `return route(project_root)` without a state advance**, that's not a bug waiting to happen — it IS a bug. Fix it as soon as you see it.
- **Cross-reference:** the `ALIGNMENT_CONFIRMED` branch at `src/unit_14/stub.py:1531-1534` is the correct pattern; the `ALIGNMENT_FAILED` branch was the regression. The two branches should be structurally identical in their recursion-safety: advance state, save state, recurse.
- **Out-of-scope follow-up:** audit ALL `return route(project_root)` call sites for similar bugs. If any others are found, file them as S3-115+. **UPDATE (Bug S3-115):** audit completed. 15 recursive routing sites inspected. Zero additional defects found. S3-114 was a genuine one-off regression.

### Lesson: Recursive Routing Invariant Locked In By Parametrized Tests (Bug S3-115)

- **Follow-up to Bug S3-114.** After fixing the routing infinite-recursion defect, the lessons learned entry flagged an audit of every recursive routing call site. Bug S3-115 executed that audit and the conversion of its findings into a permanent test-enforced invariant.
- **Audit outcome: 15 recursive routing sites, zero additional defects.** 12 direct `return route(project_root)` + 2 `_route_debug` self-recursions + 1 `_route_stage_3` delegation. Every site is already preceded by a state-advancing operation (`advance_sub_stage`, `advance_stage`, `update_debug_phase`, `increment_*`) + `save_state`. S3-114 was a one-off regression, not a recurring pattern.
- **Why the audit was still productive despite zero findings:** the recursion-safety invariant was implicit — a contributor had to notice the ALIGNMENT_CONFIRMED pattern and replicate it. The S3-114 contributor failed that replication by accident. The audit's real value is converting the implicit invariant into an explicit one — a parametrized test class where adding a new recursive routing branch requires adding a corresponding test row.
- **Rejected alternatives:**
  - **Bounded-recursion decorator** (`@bounded_route(max_depth=10)`) wrapping every routing function. Runtime guard. Over-engineering: catches the symptom without communicating the invariant; adds routing overhead; has design costs (what depth? propagate through fixtures? false positives on legitimate long chains). Rejected because the audit found only one past occurrence — one bug does not justify a runtime guardrail class.
  - **Runtime depth tracking** (counter argument passed through recursive calls). Same problems as the decorator. Rejected.
  - **Static analysis check** (AST walk asserting every `return route(project_root)` is preceded by a `save_state` call). Interesting but brittle (hard to prove the preceding call graph advances state); not worth the effort for one occurrence.
- **Fix: test class `TestRoutingRecursionBoundedness` (parametrized, 11 rows) + `TestRouteDebugRecursionBoundedness` (3 tests).** Each test sets up state for the target branch, writes the trigger `last_status`, wraps `route(project_root)` in `sys.setrecursionlimit(100)`, and asserts route returns a valid action block without `RecursionError`. The bounded `setrecursionlimit(100)` is the enforcement mechanism: a broken recursion path hits the limit and raises, making the failure deterministic and loud.
- **Prevention rule:** every new `return route(project_root)` (or equivalent recursive routing call) MUST be accompanied by a new row in `TestRoutingRecursionBoundedness`, or a dedicated test if the setup is idiosyncratic. Contributors who don't add a row get no regression test; reviewers should flag this in PR review.
- **Meta-pattern — "When a defect is a one-off in a structural invariant, convert the invariant from implicit to explicit with tests — not runtime guards."** Runtime guards (decorators, depth checks) catch the symptom but don't communicate why the rule exists. Parametrized tests with one row per call site force the rule to be re-asserted every time the rule expands. The test class IS the invariant enforcement point, and failure messages direct the reader to the rule rather than a stack trace.
- **Test-driven invariant preservation is strictly better than runtime guardrails in this case because:**
  1. Tests are documentation — the SITES table lists every recursive routing branch, making the audit permanent and discoverable.
  2. Tests fail at CI time, not at runtime in production. Stack-overflow crashes in production are much worse than test failures in CI.
  3. Tests force the contributor to acknowledge the invariant when adding new code — a decorator lets them silently introduce a broken branch that the runtime will catch too late.
  4. Tests do not add any runtime overhead.
- **When should you use a runtime guard instead?** When the invariant is reachable by inputs the test suite cannot enumerate (e.g., user-provided data with unbounded variation). Routing branches are a finite, enumerable set — parametrized tests cover the whole space.

### Lesson: Deterministic Two-Sided Validation for LLM-Generated Structured Artifacts (Bug S3-116)

- **Bug:** S3-116 (reported externally). A real SVP project's blueprint used em-dash separators (`## Unit 1 — Plugin Scaffold`) instead of the canonical colon (`## Unit 1:`). The framework's strict regex `^## Unit (\d+):` returned zero matches, `_count_unit_headings` returned 0, and `run_infrastructure_setup` raised the cryptic `"No unit headings found in blueprint. Cannot proceed with setup."` — with no diagnostic about which files were scanned, what format was expected, or what the blueprint actually contained. The operator debugged by instrumenting the parser.
- **The spec had the grammar documented at Section 1949** ("Both files use the same unit heading structure `## Unit N: Name`") but nothing enforced the rule deterministically until the parser silently failed at Stage 3. Documentation without enforcement is a guideline, not an invariant.
- **Three layers of silent failure:**
  1. No early deterministic validation — `blueprint_checker` is an LLM agent, its checklist didn't require format validation, the blueprint could pass all of Stage 2 with em-dash headings.
  2. Silent parser output — `_count_unit_headings` (Unit 11), `extract_units` (Unit 8), and `parse_signatures` (Unit 9) all returned empty results with no near-miss detection or warning.
  3. Weak downstream error — the Stage 3 ValueError had zero diagnostic content.
- **Fix — deterministic two-enforcement-point validation with a shared function:**
  1. **Shared validator** `validate_unit_heading_format` in Unit 8 is the single source of truth. Uses a loose pattern (`^## Unit \d+\b`) to detect candidate headings, then filters to those NOT matching the strict canonical pattern (`^## Unit \d+:`). Returns a list of `(filename, line_number, line_text)` near-miss tuples. Empty list = format-clean. Paired with `format_unit_heading_violations` for consistent error messages.
  2. **Writing side**: `dispatch_agent_status` for `blueprint_author + BLUEPRINT_DRAFT_COMPLETE` / `BLUEPRINT_REVISION_COMPLETE` (Unit 14) calls the validator and raises `ValueError` on violations. Pipeline halts IMMEDIATELY after blueprint_author completes, BEFORE Gate 2.1 is presented. The operator sees the violation with full near-miss diagnostic, can re-invoke the agent with feedback, and doesn't waste gate cycles approving a malformed blueprint.
  3. **Extraction side**: `run_infrastructure_setup` Step 5 (Unit 11) calls the same validator as a safety net. If the writing-side dispatch check was skipped (operator bypass, dispatch regression), the extraction-side check catches it with the same diagnostic.
  4. **Agent definition updates**: `BLUEPRINT_AUTHOR_DEFINITION` and `BLUEPRINT_REVIEWER_DEFINITION` (Unit 20) explicitly reference the grammar rule and note that dispatch halts the pipeline on violation. LLM-level nudge + deterministic runtime enforcement = belt and suspenders.
- **Pattern — "A deterministic parser must never silently return empty for a structured document without reporting what it actually found."** When a regex returns 0 matches on a non-empty input, the parser must either (a) fall back to a loose pattern and surface the near-miss in an error, or (b) raise with file/line context. Silent empty return is the wrong default for any parser operating on a human-written document.
- **Pattern — "Documented grammar requires enforcement infrastructure, not just prose."** Spec Section 1949 documented the rule all along. Documentation is not enforcement. Bug S3-116 added the missing enforcement infrastructure (validator + two dispatch points) so the spec rule is backed by runtime checks.
- **Pattern — "Validate LLM-generated structured artifacts at the earliest-possible dispatch point."** Combined with the Bug S3-112 lesson ("agent discretion on deterministic facts is a liability"), the rule becomes: when an LLM agent produces a structured artifact, every grammar rule the parser depends on must be enforced deterministically at the earliest dispatch step (typically `dispatch_agent_status` after the agent emits its terminal status). The parser is a consumer of the grammar; the dispatch is the enforcement point of the grammar.
- **Prevention rules:**
  1. **Every parser that enforces a strict grammar on a human/LLM-written document MUST have a near-miss detector.** When strict parsing returns empty, run a loose pattern and surface what was found in the error message.
  2. **Every grammar rule documented in the spec MUST have at least one deterministic runtime check.** If the spec says "do X", something deterministic must check that X was done. Prose is not enforcement.
  3. **Error messages for parser failures MUST include** (a) which files were scanned, (b) the expected format with an example, (c) the actual near-miss content found, (d) a cross-reference to the spec section documenting the grammar, (e) the bug ID for operators to look up the fix rationale.
  4. **When adding a new dispatch layer for LLM output validation, share the validator with the downstream consumer** so the two sides cannot drift. The writing side and the extraction side of S3-116 call the same Unit 8 function.
  5. **LLM agent definition updates are belt-and-suspenders, not a substitute for deterministic checks.** The agent definition updates in Unit 20 reduce the probability of violations but do not guarantee their absence. The dispatch-level check is the real guard.
- **Cross-reference:** S3-112 (agent discretion on deterministic facts) and S3-111 (generator formula drift) are siblings in the "silent wrong output from LLM agents" family. S3-116 is the parser-level analog — silent wrong output from a parser consuming LLM input.

### Lesson: Deterministic CLIs Belong in Action Blocks, Not Orchestrator Heads (Bug S3-117)

- **Bug:** S3-117 (reported externally). During a real Stage 2 → Stage 3 transition, the orchestrator received a `run_command` action block carrying only a semantic command name (`stub_generation`) and had to spend ~15 rounds of grep/read/guess discovering the concrete CLI (`python scripts/stub_generator.py --blueprint ... --unit N --output-dir ... --upstream "..." --language ...`). The command-to-CLI mapping is 100% deterministic from `(command_name, state, profile)`, but the action block didn't carry the mapping — the orchestrator had to reconstruct it each time.
- **Context — asymmetric protocol.** The parallel action block types already carried concrete execution strings:
  - `invoke_agent` had a `prepare` field with the concrete `python scripts/prepare_task.py --agent X --project-root . --output .svp/task_prompt.md [--unit N]` CLI (built by `_agent_prepare_cmd`).
  - `human_gate` and `run_command` had a `post` field with the concrete `python scripts/update_state.py --command X --project-root .` CLI.
  - Only `run_command` was missing its concrete **execution** CLI. It had a semantic `command` name but no `cmd` field saying HOW to execute it.
- **Fix:** new `cmd` field in `run_command` action blocks. Populated by four private helper functions in Unit 14 (`_cmd_stub_generation`, `_cmd_quality_gate`, `_cmd_test_execution`, `_cmd_compliance_scan`), each deterministic from `(state, project_root)`. The orchestrator runs `cmd` verbatim via its Bash tool. Applied to seven script-based routing emitters. Four semantic operator commands (`lessons_learned`, `debug_commit`, `unit_completion`, `stage3_reentry`) have no script CLI and keep `cmd` absent; the orchestration skill documents that the orchestrator follows the `reminder` text for those cases.
- **Pattern:** same root-cause shape as S3-112 (agent discretion on deterministic facts), S3-111 (generator formula drift), S3-116 (deterministic enforcement on both sides). **A deterministic fact was left for someone else to reconstruct.** S3-117 applies the lesson to the orchestrator instead of the LLM agents or the blueprint parsers: *whoever executes a command should not have to guess how to execute it.*
- **New prevention rules:**
  1. **If an action block says "do X", the action block must contain the concrete instructions to do X.** No semantic-name lookups, no command-to-CLI mapping tables in the orchestration skill. Concrete CLIs only, built by deterministic helpers from state.
  2. **Asymmetric protocols are structural bugs.** `invoke_agent` had `prepare`, `human_gate` had `post`, but `run_command` had no concrete execution field. Whenever one protocol branch has more detail than a parallel branch, the less-detailed branch is usually the bug. Audit for symmetry.
  3. **Orchestrator-side research during normal operation is a protocol smell.** If the orchestrator has to grep, read source files, or guess during normal (non-debug) operation, the protocol output is incomplete. The fix is not to document the missing information in the skill — it's to add the information to the protocol output.
  4. **The orchestration skill should not contain "how to run command X" instructions.** The skill documents the six-step cycle, the break-glass protocol, and the action type handling rules. It should NOT contain command-to-CLI mapping tables. Instructions in the skill accumulate and rot; instructions in action blocks are recomputed every routing cycle and stay fresh.
- **Cross-reference:** sibling to S3-112 in the "deterministic fact left for someone else to reconstruct" family. Sibling to S3-116 in the "validator/builder lives in one place and is called at every point of use" family. Completes the pattern: for every place where the pipeline emits a structured artifact (action block, parser output, agent prompt), the artifact must carry the concrete executable information — not a semantic hint that the consumer must expand.

## Relative Paths at CLI Boundaries Are Structural Hazards (Bug S3-118)

- **Bug.** During the first unit cycle of Stage 3 in a real consumer project (debrief1.0), the quality gate crashed with `EnvironmentLocationNotFound: Not a conda environment: /Users/cfusco/anaconda3/envs/svp-`. The env name was literally `svp-` (empty suffix). The orchestrator then wasted six rounds of `python -c` probe one-liners diagnosing and ultimately started hand-fixing E501 test residuals directly — a protocol violation it resorted to because the broken quality-gate status had no obvious dispatch path.
- **Root cause (two-layer).**
  - *Layer 1.* `derive_env_name(project_root)` in `src/unit_1/stub.py:81` returned `f"svp-{project_root.name}"`. `Path('.').name` is the empty string. Relative project-root → nonsense env name → conda failure. The helper had no defensive resolve.
  - *Layer 2.* Asymmetric `--project-root` handling across CLI `main()` functions. Unit 11 (`setup_env`) and Unit 14 (`routing`, `update_state`, its `prepare_task` bundle) already did `Path(args.project_root).resolve()` at parse time. Unit 15's `run_quality_gate_main` and Unit 13's `main` (the `prepare_task` CLI) did not. Because S3-117's new `_cmd_quality_gate` helper emits `--project-root .`, the unresolved `Path('.')` flowed straight from argparse into `derive_env_name` and the failure fired deterministically on every quality-gate invocation in a consumer project. Half the codebase was defensive, half was sloppy, and nothing enforced the convention.
- **Fix.**
  1. **Layer 1 — defensive resolve in the helper.** `src/unit_1/stub.py:82` now returns `f"svp-{project_root.resolve().name}"`. One-line fix that repairs every caller regardless of input shape. `.resolve()` is a no-op on already-absolute paths; cost is one `stat()` per call.
  2. **Layer 2 — resolve at the CLI boundary.** `src/unit_15/stub.py:365` and `src/unit_13/stub.py:1943` now do `Path(args.project_root).resolve()`. Brings them into alignment with Unit 11 and Unit 14.
  3. **Convention lock — AST regression test.** New `tests/regressions/test_bug_s3_118_project_root_resolve.py::test_all_cli_main_functions_resolve_project_root` walks every `src/unit_*/stub.py`, finds every `project_root = <expr>` assignment whose RHS mentions `args.project_root`, and asserts the expression is a `.resolve()` call. Works regardless of enclosing function name (Unit 15's is `run_quality_gate_main`, not `main`). If any future CLI script drifts, the test fails with the exact file and line number.
- **Pattern family.** Sibling to S3-111 (generator formula drift), S3-112 (agent discretion on deterministic facts), S3-116 (two-sided validation with a shared validator), and S3-117 (asymmetric protocols are structural bugs). Same root-cause shape applied to a different surface: **when half of the codebase is defensive about a deterministic invariant and the other half isn't, the un-defensive half is a bug waiting for its trigger.** The convention existed informally in Unit 11 and Unit 14, but "informally" means "not tested", which means "Unit 15 drifted silently and nobody noticed until a consumer project hit Stage 3."
- **New prevention rules:**
  1. **Resolve untrusted inputs at protocol boundaries, once.** A `Path` value is like user input — a relative path is syntactically legal but semantically degenerate (`Path('.').name == ''`). Resolve once at the CLI boundary; downstream code then trusts that `project_root.name` is meaningful.
  2. **Defensive internal helpers cost microseconds.** `derive_env_name` could trust its callers or defensively resolve. Trust failed; resolve costs nothing. Prefer defensive helpers for invariants that would blow up deep in the stack — the defensive call is obvious from the source, and the failure mode it prevents is not.
  3. **Conventions are only real when a test enforces them.** The "every CLI `main()` resolves" convention existed in Unit 11 and Unit 14, but nothing prevented Unit 13 and Unit 15 from drifting. AST-walk tests are cheap convention locks: walk the source, find the pattern, assert the shape. Generalizes to any cross-file invariant. **This is the first SVP test that locks a convention via AST walk across every unit; the pattern will be reused.**
  4. **Asymmetry is a structural smell.** If one half of the codebase has a defensive pattern and the other half does not, the non-defensive half is a latent bug. Audit for symmetry whenever you touch the defensive half.
- **Cross-reference:** the third entry in the "asymmetric conventions on deterministic invariants are bugs" family. S3-117 applied this to action block schemas (concrete `cmd` vs semantic names); S3-118 applies it to CLI script conventions (`.resolve()` at parse time). Whenever you find an asymmetric defensive pattern, look for the twin bug on the undefended half.

## Blueprint Self-Review Moves Structural Checks Upstream (Enhancement, not a bug)

- **Insight.** Downstream cold review (Blueprint Reviewer, Alignment Checker) catches structural issues at significant latency. The same checks can run in the blueprint author step if the author has a concrete checklist. Six universal categories — schema coherence, function reachability, invariant coherence, dispatch completeness, branch reachability, contract bidirectional mapping — apply to any blueprint regardless of project archetype or language.
- **Why it matters.** Structural bugs caught at the author step cost a revision loop. Caught at the alignment checker, they cost a full review cycle. Caught at implementation time, they cost stubs and tests. Caught at integration, they cost a re-derivation from the spec. Cost grows roughly an order of magnitude per stage.
- **Design.** The blueprint author definition (Unit 20) is expanded with a new "Self-Review Pass" methodology step requiring the author to fill `.svp/blueprint_self_review.md` with PASS/FAIL evidence for every item in `.svp/blueprint_author_checklist.md` and **iterate until ALL_PASS** before emitting the terminal status. The Checklist Generation Agent (Unit 23) is expanded to embed the six categories from spec Section 44.11 into the generated checklist, refined per the project's primary language. **Routing is deliberately untouched** — the self-review is a behavioral contract on the agent, not a deterministic dispatch check. The downstream cold reviewers continue to run as a safety net.
- **Why agent-prompt-only enforcement.** Adding more dispatch-level enforcement points increases routing surface area and risk. The S3-116 unit heading check is the only deterministic dispatch enforcement for the blueprint author and we have no appetite to add a second. The self-review is enforced by the agent's own prompt, with the persisted artifact as audit trail. If an agent skips the self-review, the downstream Blueprint Reviewer and Alignment Checker still catch issues — at higher cost, but correctness is preserved.
- **Pattern family.** Sibling to S3-116 (deterministic invariants need explicit enforcement points), S3-117 (concrete instructions belong in the artifact, not in the consumer's head), S3-118 (conventions are only real when tested). The new pattern: **producer-side self-review with a concrete artifact catches structural issues at the cheapest point.**
- **Prevention rules:**
  1. **Self-review is cheapest at the producer.** When the producer has full context (the dialog, the decomposition rationale), it is the best-positioned actor to catch structural issues. Downstream reviewers catch fewer issues per minute of reading because they have less context.
  2. **Universal structural principles are project-agnostic.** Schema coherence, reachability, invariants, dispatch completeness, branch reachability, and contract mapping apply to every blueprint. There is no archetype or language that is exempt — only the concrete check format varies, which is what the Checklist Generation Agent's per-language refinement handles.
  3. **A filled self-review artifact beats a prose instruction.** "Check your work before submitting" is unverifiable. "Write a self-review file with each item marked PASS or FAIL with a citation" is verifiable. Make structural checks produce artifacts, not just internal beliefs.
  4. **Some enforcement belongs in agent prompts, not in dispatch.** Not every check needs to be a deterministic gate. When the cost of agent skipping is bounded (downstream reviewers exist), agent-prompt-only enforcement reduces routing surface area and avoids the regression risk of adding new dispatch checks. Use the durable artifact as the audit trail.
- **Cross-reference:** the first explicit "agent-prompt-only enforcement is enough when downstream cold review exists" entry in the lessons learned. Future enhancements that consider adding dispatch-level checks should compare against this entry — if downstream review catches the failure mode, prompt-level enforcement may be the right scope.

## Slash Command Definitions That Bootstrap Session State Must Be Thin State Triggers (Bug S3-119)

- **Bug.** A user invoked `/svp:bug` from a Claude Code plugin build (`debrief`) to triage a blueprint regression (SKILL.md / agent .md frontmatter missing). The bug triage agent ran correctly, wrote `.svp/triage_result.json`, and returned. Routing then silently refused to advance. The user's report traced the failure through `route()`, `_route_debug`, `dispatch_gate_response("gate_6_0", "AUTHORIZE DEBUG")`, and `update_state.py --phase bug_triage` and demonstrated that every one of those code paths is guarded by `state.debug_session is not None` — but no code path creates `debug_session` when `/svp:bug` is invoked cold from pipeline_complete. The only pre-existing post-delivery bootstrap was `gate_pass_transition_post_pass2 + FIX BUGS`, a one-shot gate. After that gate has been consumed, `/svp:bug` had no re-entry path.
- **Root cause.** Spec §12.18.13 (Debug Phase Transition Summary Table) mandates `/svp:bug entry → triage: null → "triage"` creating the `debug_session` object with `authorized: false`. No code implemented the transition. The prior `_SVP_BUG_DEFINITION` documented a Group B 5-step action cycle (`prepare_task → agent → status → update_state --phase → routing`), none of which create state. The 5-step pattern is correct for **stateless** Group B commands (`/svp:help`, `/svp:hint`, `/svp:ref`, `/svp:redo`) because those commands dispatch an agent that reads state without needing to create session state first. `/svp:bug` is fundamentally different: it bootstraps a new session. But its definition was written as if it were a stateless command.
- **Fix.** Two-file change mirroring the `/svp:oracle` thin-trigger pattern introduced in Bug S3-79:
  1. **`src/unit_14/stub.py` `dispatch_command_status`** — new `svp_bug_entry` branch next to `oracle_start`. It calls `enter_debug_session(state, 0)` (bug_number 0 is the placeholder convention already used by other bootstrap sites; the triage agent assigns the real number from `len(debug_history) + 1`). Two guards: refuses if `oracle_session_active` is True (spec §35.3 — `/svp:bug` is blocked during oracle sessions; oracle enters debug internally via Gate 7.B), refuses if `debug_session is not None` already (spec §12.18.1 — one debug session at a time).
  2. **`src/unit_25/stub.py` `_SVP_BUG_DEFINITION`** — rewritten as a thin trigger. Step 1 runs `update_state.py --command svp_bug_entry`, Step 2 runs `routing.py`, Step 3 says "follow the six-step action cycle from there". Routing then emits Gate 6.0, `dispatch_gate_response` authorizes, routing dispatches the bug_triage_agent via the existing `_route_debug` machinery, and the inline triage-result loader at routing.py:1201-1215 (Bug S3-84 fix) loads `triage_result.json` and advances to Gate 6.2. No changes needed to `_route_debug` itself.
- **Detection.** External user report during a non-SVP project build. The pipeline's normal test suites did not catch this because the only integration-tested bootstrap path (`pass_transition + FIX BUGS`) happens to work. The `/svp:bug` slash command itself was never dispatched through state during regression testing — only its agent prompt assembly was tested. **The skill definitions are code that never gets executed by pytest.** Anything encoded in a skill definition is effectively outside the test envelope unless a test parses the skill definition and simulates its action cycle against `dispatch_*` functions.
- **Pattern.** P10 (Error-Path Contract Omission), same as Bugs S3-84 (triage result not loaded into debug session) and S3-86 (PHASE_TO_AGENT mapping mismatch). The shape is: **the spec mandated a state transition, the code implemented every consumer of that state, but nothing implemented the producer.** Every consumer gets guarded against the null case, so the failure mode is a silent refusal to advance, not a crash — which makes it invisible until a user hits it and traces every code path manually.
- **New prevention rules:**
  1. **Slash command definitions that bootstrap stateful sessions MUST be thin state-transition triggers.** They run exactly one `update_state.py --command <entry>` and then redirect to `routing.py`. Everything else — agent invocation, status dispatch, gate presentation — is the routing script's job. The five-step Group B pattern is for *stateless* commands only. The invariant at Unit 25 (originally Bug S3-79) is extended by this bug to explicitly name `/svp:bug` alongside `/svp:oracle` as a thin trigger.
  2. **Every state transition mandated by the spec must have a regression test that invokes the producer, not just the consumers.** The S3-84 fix added tests that verified `_route_debug` loads `triage_result.json` — a consumer. Nothing tested that `_SVP_BUG_DEFINITION` actually produces the state transition it was supposed to produce. When a bug report says "the spec says X happens but it doesn't", look for the producer, not just the consumer.
  3. **Skill definitions are executable by Claude Code but untested by pytest.** Any behavioral contract encoded in a `*_DEFINITION` string constant is outside the test envelope unless a test (a) parses the definition, (b) simulates its action cycle against the relevant `dispatch_*` functions, and (c) asserts the expected state transitions. The S3-119 regression test suite introduces Test E (parses `_SVP_BUG_DEFINITION`, asserts it contains `svp_bug_entry` and does not contain `prepare_task.py`) — a lightweight pattern that locks the thin-trigger shape. Extend to `/svp:oracle` and future thin-trigger commands.
  4. **When adding a new `Group B` slash command, decide upfront whether it bootstraps session state.** If yes, use the thin-trigger pattern (command → routing). If no, use the five-step action cycle (prepare → agent → status → update → routing). Document the choice at the skill definition site. Do not silently apply the stateless pattern to a stateful command.
- **Cross-reference:** Sibling to Bugs S3-84, S3-86 in the "P10 Error-Path Contract Omission — producer missing while every consumer is implemented" family, and to Bug S3-79 in the "slash command skill definitions must be thin triggers" family. Unlike S3-79 (which was about content construction belonging in routing), S3-119 is about **state transitions** belonging in routing. The two lessons combine into: skill definitions must be content-free **and** state-free — they exist only to hand off to the routing script.

## Gate Response Dispatches Must Honor Every Column of the Spec Debug Phase Transition Table (Bug S3-120)

- **Bug.** During a `/svp:bug` session on a downstream plugin (`debrief`), after the user picked FIX BLUEPRINT at Gate 6.2, the orchestrator entered an infinite loop re-invoking the triage agent. Dispatch set `stage=2, sub_stage=blueprint_dialog` correctly, but `debug_session.phase` remained `triage`. The routing priority at the top of `route()` ("Debug session routing takes highest priority") sends any non-null `debug_session` to `_route_debug`, which sees `phase=triage` and re-invokes the triage agent. FIX SPEC had the symmetric defect.
- **Root cause.** Pattern P7 (Spec Omission in Blueprint). Spec §12.18.13 (Debug Phase Transition Summary Table) explicitly mandates for the Gate 6.2 FIX BLUEPRINT and FIX SPEC rows: `"repair" → session terminated. debug_session: null (via abandon). Delete triage_result.json. All counters reset.` The blueprint contract list entries for these two responses omitted the `abandon_debug_session` and `triage_result.json` deletion clauses — they listed only `version_document` and `restart_from_stage`. The Unit 14 implementation faithfully followed the incomplete blueprint. Same P7 failure class as S3-84 (triage result not loaded into debug session state) and S3-99 (dispatch contract omitted the triage result load).
- **Fix.** Amend the blueprint contract lines at `gate_6_2_debug_classification + FIX BLUEPRINT` and `+ FIX SPEC` to enumerate every column of the spec table row (version_document, abandon_debug_session, delete triage_result.json, restart_from_stage). Add the abandon + unlink steps to the corresponding branches of `dispatch_gate_response` in `src/unit_14/stub.py`. Reuse `abandon_debug_session` (already imported, already used in four other dispatch branches) and the existing `.svp/triage_result.json` unlink pattern from the `stage3_reentry` handler. No new primitives.
- **Regression test.** New file `tests/regressions/test_bug_s3_120_gate62_fix_blueprint_dangling_debug.py`. The canonical guard is the test that seeds a `debug_session` with `phase=triage`, dispatches FIX BLUEPRINT, saves state to disk, re-enters `route(project_root)`, and asserts the returned action is NOT a `bug_triage_agent` invocation. This is the test that would have caught the original bug before it shipped — it directly tests the loop closure, not just the individual state transition. Over-correction guards confirm FIX UNIT and FIX IN PLACE still preserve `debug_session` (they are the branches that legitimately keep the session alive).
- **Detection.** The orchestrator caught it in a debrief. The existing `test_bug55_rollback_gate62_wiring.py` regression test covered FIX UNIT at Gate 6.2 but had no coverage for FIX BLUEPRINT or FIX SPEC. The test envelope was not complete — the FIX UNIT branch was tested because Bug 55 was about rollback wiring, but FIX BLUEPRINT and FIX SPEC were never exercised through dispatch despite being different code paths subject to the same spec table.
- **Pattern.** P7 (Spec Omission in Blueprint). The spec table has four columns (Phase change, State mutation, Artifact write/delete, Counters). The blueprint contract dropped two of them. The code matched the contract, not the spec. Classic P7: the translation from spec to blueprint lost information.
- **New prevention rule:** **When a gate/response dispatch touches a row of the spec §12.18.13 Debug Phase Transition Summary Table, the blueprint contract line MUST enumerate every non-null column of that row in order.** A blueprint contract that reads "X, Y" when the spec row specifies four actions is a structural defect. A future blueprint-checker invariant could lock this in by parsing the spec table and, for every row, asserting the corresponding blueprint contract line names every required function and every required artifact action. Not implemented as part of S3-120 because it would extend blueprint-checker scope; flagged as a follow-up in Section 24.133 of the spec.
- **Cross-reference:** Sibling to Bugs S3-84 and S3-99 in the P7 "Spec Omission in Blueprint" family at Gate 6.2. The cumulative lesson across these three bugs is that **the Section 12.18.13 transition table is the single source of truth for debug-session dispatch contracts**, and the blueprint must treat each row as a four-column contract to copy, not a one-column summary to paraphrase.

## S3-121: Plugin Command Filename Double-Prefix

- **Bug.** The SVP plugin shipped 11 command files named `svp_bug.md`, `svp_save.md`, `svp_oracle.md`, etc. Claude Code's plugin loader registers slash commands as `{plugin_namespace}:{filename_stem}` — prepending the namespace from `plugin.json` and using the filename stem verbatim, with no prefix stripping. The SVP plugin declares `name: "svp"`, so the loader registered these files as `/svp:svp_bug`, `/svp:svp_save`, `/svp:svp_oracle`, etc. — **double-prefixed**. Meanwhile every document in the repository (spec, blueprint, agent prompts, lessons learned) and the `# /svp:bug` markdown heading inside each file body correctly cited the bare form `/svp:bug`, `/svp:save`, etc. The documented form never matched the registered form.
- **Why it wasn't caught for so long.** Three reinforcing factors. (1) Users click through Claude Code's autocomplete list rather than typing the full `/svp:svp_bug` form, so the awkward double-prefixed name was accepted rather than questioned. Autocomplete papers over filename drift. (2) Every document used the correct `/svp:<name>` form, which matched the `# /svp:<name>` body headings inside the files, creating an internally consistent documentation story that looked correct at every surface except the actual registration. (3) S3-87 diagnosed the inconsistency between `svp:orchestration` and `svp:svp_bug` in the wrong direction: it edited the skill frontmatter `name` to `"svp:svp_orchestration"` to "align" the skill with the commands, instead of recognizing the commands as the broken target. That edit was also **functionally inert** — skill registration is driven by the directory path `skills/orchestration/` → `svp:orchestration`, not by the frontmatter `name` field, which is documentation. S3-87 aligned a documentation field to a broken target without fixing anything.
- **Cross-project corroboration.** A sibling project (`debrief`) copied SVP's layout as a canonical reference, hit exactly this bug independently, and documented it in its own Bug Catalog as BUG-AUDIT-10. That diagnosis was relayed back to this project as an advisory on 2026-04-14. A `claude-code-guide` subagent independently verified the diagnosis against the official Claude Code Plugins Reference documentation. Most illuminating: the docstring of our own `tests/regressions/test_bug_s3_87_skill_name_prefix.py` already stated the rule correctly — *"Claude Code derives command names from {namespace}:{filename_stem}, so svp_bug.md becomes svp:svp_bug"* — but the assertions in that file codified the wrong conclusion from the correct rule. A correct premise had been written down and then ignored.
- **Fix.** Four coordinated changes. (a) Rename all 11 command files to bare names via `src/unit_25/stub.py` `COMMAND_NAMES` list and `COMMAND_DEFINITIONS` dict keys — `svp_help`→`help`, `svp_hint`→`hint`, `svp_ref`→`ref`, `svp_redo`→`redo`, `svp_bug`→`bug`, `svp_oracle`→`oracle`, `svp_save`→`save`, `svp_quit`→`quit`, `svp_status`→`status`, `svp_clean`→`clean`, `svp_visual_verify`→`visual-verify` (hyphen, matching the spec §35.18 documented form). (b) Invert the S3-87 edit in `src/unit_26/stub.py` orchestration skill frontmatter: `svp:svp_orchestration` → `svp:orchestration`, matching the directory-derived registration. (c) Defensive cleanup in `src/unit_23/stub.py` plugin assembly: unlink any stale `*.md` files in `commands_dir` before writing the fresh set, so a cached `svp_bug.md` cannot survive reassembly and continue to register alongside the new `bug.md`. (d) Regression guard test `tests/regressions/test_bug_s3_121_command_double_prefix.py` with a negative sentinel (no key in `COMMAND_DEFINITIONS` starts with `svp_`), a positive regex guard (`^[a-z][a-z_]*$`), and a body-heading-vs-filename drift check. The S3-87 regression test is renamed to `test_bug_s3_121_skill_name_correction.py` and its assertions are inverted to assert the bare skill name. The S3-69 hyphen guard is preserved.
- **Lessons.**
  1. **`{namespace}:{stem}` with no prefix stripping.** This is the Claude Code plugin loader's registration rule for slash commands. Internalize it: filenames live in the `{stem}` position of a `{namespace}:{stem}` template. Any prefix baked into the filename is additive, not replacive. If the plugin namespace is `foo`, a file named `foo_bar.md` registers as `/foo:foo_bar`, not `/foo:bar`.
  2. **Markdown headings in file bodies are documentation only.** The `# /svp:bug` heading at the top of each command file has zero effect on how Claude Code registers the command. It is purely for users reading the file. Do not use in-body headings as evidence that filenames are correct — they are orthogonal concerns.
  3. **Skill names derive from the directory path, not the frontmatter `name` field.** A skill at `skills/orchestration/SKILL.md` registers as `<namespace>:orchestration` regardless of what the frontmatter says. The frontmatter `name` field is documentation that should match reality. If you see a mismatch between frontmatter and registered name, fix the frontmatter, not the directory path — unless the directory path is itself wrong, in which case fix the directory.
  4. **Cross-project corroboration is a valid and load-bearing detection mode.** A sibling project building against our layout as a canonical reference hit this bug independently and relayed the diagnosis back. Treat advisories from projects that use SVP as a reference implementation as high-signal. Preserve this practice deliberately: when downstream projects report a bug, investigate the canonical source even if local tests pass.
  5. **The S3-87 mis-diagnosis pattern: aligning a text field to a broken target instead of fixing the target.** When two things are inconsistent, the reflexive move is to make them match. The question to ask first is: *which one is correct?* If you cannot answer that question confidently, do not edit either. S3-87 aligned a skill frontmatter field to the broken commands because the commands were the larger set and felt authoritative. The correct move would have been to check the docs and recognize the commands as the bug. A useful diagnostic protocol: when fixing an "inconsistency", write down on one side what the correct value should be and on the other side what the broken value is. Do not proceed until you have a documented reason why one side is correct.
  6. **Negative sentinels guard against regressions the positive tests cannot see.** The S3-87 positive test asserted `svp:svp_orchestration is present`, which passed because the broken value was there. A negative sentinel (`svp:svp_orchestration is NOT present`) is what the S3-121 corrected test asserts, and is what catches a regression if someone re-introduces the double-prefix in the future. Every rename should grow a negative sentinel test of the form "the old name is forbidden", not just a positive test of "the new name is present".
  7. **User-side cache invalidation after a filename rename.** Claude Code caches resolved plugin contents at `~/.claude/plugins/cache/svp`. After pulling a fix that renames command files, users must clear that cache or the session will continue to see the old double-prefixed commands alongside the new bare-name commands. Two options: `claude plugin uninstall svp@svp && claude plugin install svp@svp`, or `rm -rf ~/.claude/plugins/cache/svp`. Document this prominently in the bug entry and in release notes — users will otherwise think the fix didn't work.
- **Pattern.** P7 (Specification Omission). The spec did not document the Claude Code registration rule — specifically, that the plugin namespace is prepended to the filename stem with no prefix stripping. Without that rule in the spec, S3-87's author aligned the skill frontmatter to the broken target. The absence of a written registration rule was the upstream cause of both S3-87's mis-diagnosis and S3-121's existence. Pattern P7 manifests here as a **platform contract omission**: the spec documented SVP's internal contracts exhaustively but did not document the contracts with the hosting platform (Claude Code) that SVP depends on. Consider a subfamily of P7 for "host platform contract omission" going forward.
- **Cross-reference.** Inverts S3-87 (Orchestration Skill Name Missing svp_ Prefix). Sibling to S3-69 (Orchestration Skill Name Uses Hyphen Instead of Colon) — both are skill/command name registration bugs stemming from undocumented Claude Code platform contracts. The cumulative lesson across S3-69, S3-87, and S3-121 is that the Claude Code plugin loader's registration rules must be documented in the spec and enforced by regression tests; every surface (commands, skills, agents, hooks) has a distinct rule, and inferring one surface's rule from another surface's behavior leads to broken alignments. **S3-122 (next entry) extends this lesson to agents.**

## S3-122: Agent Frontmatter Name Drift (correlate of S3-121)

- **Bug.** `src/unit_23/stub.py` line 837 (and the duplicate at line 1121 in `regenerate_deployed_artifacts`) ran `name = stem.replace("_", "-")` when injecting YAML frontmatter into the 21 deployed agent files. The result: each agent file shipped with a hyphenated frontmatter `name` (e.g., `name: oracle-agent`) while its filename was underscored (`oracle_agent.md`). Claude Code's plugin loader uses the frontmatter `name` field as the registration identifier for agents (verified against the official Plugins Reference via the `claude-code-guide` subagent and confirmed by inspection of the system-reminder, which lists all 21 SVP agents with hyphenated names). So the registered `subagent_type` for each agent was `svp:oracle-agent`, `svp:bug-triage-agent`, etc. — diverging on a separator character from every internal SVP reference.
- **The drift in detail.** Every internal SVP reference uses the underscored form: `PHASE_TO_AGENT["bug_triage"] = "bug_triage_agent"` in `routing.py`, `agent_type="bug_triage_agent"` in `_make_action_block(action_type="invoke_agent", ...)` calls, `_agent_prepare_cmd("bug_triage_agent")` in the prepare command builder, `AGENT_STATUS_LINES` keys (`"setup_agent"`, `"stakeholder_dialog"`, etc.), `_agent_model_keys` mapping in unit_23, the agent files themselves, the spec, the blueprint, and the lessons learned. The orchestrator (an LLM running the orchestration skill) reads an action block saying `agent_type: "bug_triage_agent"` and must implicitly translate to `subagent_type: "svp:bug-triage-agent"` to spawn the agent via the Task tool — a translation rule that exists nowhere in the spec, the blueprint, the orchestration skill's instructions, or the code.
- **Why it hadn't crashed.** The orchestrator (the LLM) compensates for the silent producer-side transformation by performing the translation implicitly on every agent invocation. The system appears to work because a smart consumer papers over the drift. This is exactly the failure mode S3-121 corrected for commands. A literal-minded orchestrator following the action block verbatim would call `Task(subagent_type="bug_triage_agent")` and crash with `Unknown agent type`. The bug is invisible until the day the consumer stops compensating — a future runtime that strict-checks `subagent_type`, a future orchestration skill that follows action blocks more literally, an introspection tool that reports the registered name without the translation, or a different LLM that doesn't know to perform the conversion.
- **Fix.** Delete the `_` → `-` transformation. `src/unit_23/stub.py` lines 837 and 1121 change from `name = stem.replace("_", "-")` to `name = stem`. Frontmatter `name` becomes the bare filename stem (`oracle_agent` instead of `oracle-agent`); registered `subagent_type` becomes `svp:oracle_agent`; the entire chain — filename → frontmatter → registered name → action block `agent_type` → `PHASE_TO_AGENT` → `AGENT_STATUS_LINES` → `_agent_prepare_cmd` argument — uses one form throughout. No other file changes are required: every internal reference already used the underscored form. Sync regenerates the 21 deployed agent files automatically via `regenerate_deployed_artifacts()`.
- **Regression guards.** New `tests/regressions/test_bug_s3_122_agent_name_drift.py` parses the deployed agent frontmatter and asserts: (a) every frontmatter `name` value equals the filename stem verbatim, (b) no frontmatter `name` value contains a hyphen (negative sentinel), (c) every `PHASE_TO_AGENT.values()` entry corresponds to an existing agent filename stem (the routing-side reference is locked to the deployed identifier set), (d) every `agent_type` value emitted by `invoke_agent` action blocks in `routing.py` is in the set of filename stems (drift detector for the action block side). Tests import from the derived `routing` module per the S3-103 stub-import discipline.
- **Lessons.**
  1. **Frontmatter vs. directory: the loader rule depends on the surface.** For SVP plugin agents, the loader uses the frontmatter `name` field as the registration identifier. For SVP plugin skills, the directory path appears to dominate (per S3-121 empirical evidence). For SVP plugin commands, the filename stem rules. Three surfaces, three different mechanisms. **Internalize:** there is no single "Claude Code plugin loader rule" — there are several, and each has to be looked up before assuming. The Plugins Reference is the source of truth.
  2. **Generator-side identifier transformations are an anti-pattern.** When a generator transforms a source identifier (filename stem) into a different form (hyphenated frontmatter name) for a downstream consumer, the transformation creates two coexisting identifiers that can drift independently. The only way to keep them in sync is for *every* consumer to know the transformation rule — a constraint that is impossible to enforce statically. **Negative form of the lesson:** every registered identifier should be greppable in the project's source in its registered form. If the deployed registration name is `svp:oracle-agent`, then `oracle-agent` (the bare token) should appear somewhere in the project source as a literal. If it appears nowhere — as was the case for all 21 hyphenated agent names before this fix — that is a smell that the registered form is being computed by an undocumented transformation rather than declared.
  3. **Smart consumers paper over silent producers.** The reason this bug went undetected for the entire pre-2.2 lifetime of SVP is that the orchestrator (the LLM) is smart enough to translate `bug_triage_agent` → `svp:bug-triage-agent` on the fly without being told. Smart consumers are a **detection failure mode**, not a feature: every bug they paper over becomes invisible until the day the consumer changes. Any time you find that "the system works because the orchestrator is intelligent enough to figure it out", that is a defect. The orchestrator should follow the action block literally; if it cannot, the action block is wrong, not the orchestrator.
  4. **Bug families travel together.** S3-121 was about commands; S3-122 is about agents. Both are members of a larger family — "Claude Code loader convention drift" — that also includes S3-69 (skill name separator) and S3-87 (skill name double-prefix mis-diagnosis). When you find a bug in this family, immediately ask: *what other surface is this same bug going to manifest on?* The family has at least four members across three surfaces (commands, skills, agents). Hooks and the plugin manifest are the next obvious places to audit. The triage protocol after fixing one member of a family should always include "audit the sibling surfaces for the same anti-pattern".
  5. **When in doubt, ask the platform docs directly.** S3-121 and S3-122 were both verified by spawning the `claude-code-guide` subagent, which reads the official Plugins Reference. This took less than a minute per question and produced answers far more reliable than reasoning from the symptom. Any time you are about to make a structural change to how the project interacts with Claude Code, spend the minute. The cost is trivial compared to the cost of a wrong-direction fix (S3-87 is the cautionary tale).
  6. **User-side cache invalidation, again.** Same as S3-121: after pulling, users must clear `~/.claude/plugins/cache/svp` (or uninstall+reinstall the plugin) so the new frontmatter loads. Otherwise the old `svp:oracle-agent` and the new `svp:oracle_agent` may briefly coexist. Document this in the release notes prominently.
- **Pattern.** P26 (NEW — Generator-Side Identifier Transformation). The producer transforms the source identifier into a different form for the consumer. The transformation is undocumented, the result is the loader-visible name, and every other consumer references the source form. Mitigation: the transformation must either be deleted (this fix) or published in the spec AND enforced by a test that the transformation result matches every consumer's expectation.
- **Cross-reference.** Correlate of S3-121 (Command Filename Double-Prefix). Same family — Claude Code loader convention drift — different surface (agents vs commands) and different mechanism (frontmatter transformation vs filename namespace prepending). The pair confirms that the family exists and that audits of the remaining surfaces (hooks, plugin manifest, marketplace metadata, settings.json schema) should be conducted proactively rather than waiting for the next member to surface as a user-visible failure. **S3-123 (next entry) extends this family to settings.json and the scope model.**

## S3-123: Plugin Activation Leaks at User Scope

- **Bug.** SVP shipped with no mechanism for writing project-scoped plugin enablement. `svp_launcher.py:launch_session` built `subprocess.run(["claude", ...])` with no `.claude/settings.json` authoring step, so activation depended entirely on whatever was present in `~/.claude/settings.json`. Users were guided to `claude plugin install svp@svp` during setup, which defaults to `--scope user`, writing `enabledPlugins["svp@svp"] = true` into the user-scope settings file. Result: every `claude` invocation on the machine, regardless of working directory, loaded SVP. `cd /tmp && claude` still showed all `/svp:*` commands, all 21 SVP agents, the orchestration skill, and the consultant agent.
- **Why it wasn't caught.** Claude Code's plugin enablement model supports both user-scope and project-scope out of the box (documented in the Plugins Reference). SVP simply didn't adopt project-scope. During early development the user-scope default was convenient — "SVP always available" felt like a feature — and the cross-project leak consequences didn't surface until users started working on SVP and `debrief` simultaneously and found the `/plugin disable` dances unworkable. No spec section documented the Claude Code plugin loading model, so the drift between "what SVP ships" and "what the platform offers" had no anchor to catch it. This is the same upstream cause as S3-121 and S3-122.
- **Cross-project corroboration (third consecutive advisory from debrief).** `debrief` BUG-AUDIT-8 hit the bug, fixed it in commit `cb3227d`, shipped a 13-test regression suite, and relayed an advisory to SVP on 2026-04-14 with a reference implementation (`debrief/src/debrief/launcher.py:ensure_project_settings`). This is the third bug in seven days that debrief caught first and relayed back; the pattern is now a recognized SVP detection channel worth codifying as Pattern P28 (Cross-Project Advisory Loop) in a future audit.
- **Fix.** Four coordinated changes: (a) **Add Section 4.4 "Plugin Loading Architecture and Project-Scoped Activation"** to the spec — architectural primer documenting Claude Code's four config files, the enablement rule, SVP's design choice (project-scope by default), the `ensure_project_settings()` contract, and a common-mistakes checklist. This closes the upstream cause. (b) **Add `ensure_project_settings(project_root, plugin_root)` helper** to Unit 29. Idempotent, non-destructive, self-healing, corrupt-JSON-recoverable, atomic write. Writes `<project_root>/.claude/settings.json` with `extraKnownMarketplaces.svp.source.path` and `enabledPlugins["svp@svp"] = true`. (c) **Wire four call sites**: `svp_launcher.main()` new/resume/restore branches plus `_bootstrap_oracle_nested_session` in Unit 14 routing. (d) **Regression test** `tests/regressions/test_bug_s3_123_project_settings.py` modeled on debrief's BUG-AUDIT-8 suite: correctness, idempotency, preservation of unrelated keys, stale-path self-heal, corrupt-JSON recovery, atomic write, AST checks that all four call sites exist, negative assertion that no subprocess is invoked.
- **Migration (opt-in).** Existing user-scope installations continue working. To migrate, a user runs `svp` (or `svp new`/`svp restore`) in each pipeline directory to trigger `ensure_project_settings()`, then manually runs `claude plugin uninstall svp@svp --scope user` to remove the leak. Documented in `README.md` and `CHANGELOG.md`.
- **Lessons.**
  1. **Default scopes matter as much as default values.** A platform feature with multiple scopes (user vs project, global vs local) defaults to one of them, and the default leaks across contexts if wrong. Audit every platform feature SVP uses for its scope default and decide — don't inherit. Claude Code plugin enablement defaults to user scope; SVP needs project scope. Same pattern likely exists in settings.json, hook configuration, and agent tool restriction files. Future-proofing: when adopting a new Claude Code feature, write the spec section for it first (like Section 4.4 for plugin loading), name every scope the feature has, and justify the choice of default.
  2. **Self-heal in config writers is load-bearing.** The user WILL move the SVP repo on disk — different machine, external drive, reorganizing their projects directory. If the settings file hardcodes an absolute path and the helper doesn't self-heal on the next run, the pipeline silently breaks. Self-heal costs three lines of code and eliminates an entire class of "it worked yesterday" bug reports. Every config writer that persists absolute paths should self-heal when the caller provides a new path.
  3. **Atomic writes are non-negotiable for user config files.** The user may also edit the file manually. An interrupted launcher write (Ctrl-C, kernel panic, power loss) that leaves a half-written `settings.json` could corrupt the user's entire Claude Code config and is difficult to diagnose. Write-to-tmp-then-rename is five extra lines and eliminates the failure mode entirely. Never `open("w")` a user config file directly.
  4. **Visibility and operability are orthogonal — and both are required.** `ensure_project_settings()` (this fix) controls *visibility*: which plugin loads in which directory. `SVP_PLUGIN_ACTIVE=1` (set by `launch_session` in the subprocess env, gating the `non_svp_protection.sh` Bash hook from Unit 17) controls *operability*: whether Bash is permitted inside SVP sessions. A working SVP session needs both. After S3-123, a user who runs `claude` directly in a pipeline dir (bypassing the launcher) sees the plugin in autocomplete but Bash is blocked — graceful degradation. If future mitigations conflate the two mechanisms ("just check the settings file, skip the env var"), they will regress the Bash protection — do not conflate them.
  5. **Cross-project advisories from downstream plugins are high-signal — treat them as a first-class detection channel.** Three bugs in seven days were caught by `debrief` first and relayed to SVP: S3-121 (commands), S3-122 (agents), S3-123 (enablement scope). All three fixes landed in <24 hours each because the advisory pre-diagnosed the problem, the root cause, and a reference implementation. The pattern is now established: when a downstream project that uses SVP as a canonical reference reports a bug, investigate the canonical source IMMEDIATELY — do not wait for local symptoms. Consider publishing an advisory channel where downstream projects can formally relay diagnoses.
  6. **The spec is the platform contract — write it before you need it.** Sections 4.1-4.3 of the spec covered task prompt relay, hook enforcement, and cross-platform portability, but until S3-123 there was no section covering plugin loading and enablement scopes. The absence of that section was the upstream cause of S3-123 — we shipped without thinking about scope because scope wasn't in the contract. Going forward: for every Claude Code feature SVP depends on, write a Section 4.x describing the feature, its scope model, and SVP's default choice. The cost of writing a spec section before adopting a feature is an hour. The cost of discovering it wrong three months later via a cross-project advisory is a full bug-fix cycle.
- **Pattern.** P27 (NEW — Default Scope Leak). A platform feature (plugin enablement) has multiple scopes (user vs project). The default scope (user) is convenient during development but leaks the feature into unrelated contexts. The fix is to adopt the narrower scope (project) by writing the appropriate config file during project setup. Sibling of P24 (Deployed Artifact Regeneration) and P25/P26 (Generator-Side Transformation) — all four concern the gap between source-of-truth state and what the platform actually sees at runtime. Mitigation checklist: (a) spec documents every platform feature's scope model, (b) launcher writes the narrowest-appropriate scope by default, (c) regression tests verify the scope is correctly applied and the write is idempotent/self-healing.
- **Cross-reference.** Third member of the Claude Code platform-contract drift family. Members so far: S3-69 (skill name separator), S3-87 (retracted by S3-121), S3-121 (command filename double-prefix), S3-122 (agent frontmatter name drift), S3-123 (this). Next audit candidates: hook matcher syntax (hooks.json structure), agent tool restriction schema (AGENT.md `disallowedTools` field), settings.json schema validation, and marketplace metadata completeness. Proactive sweep recommended before the next cross-project advisory surfaces. **S3-124 (next entry) surfaced one of those audit candidates by accident during empirical testing.**

## S3-124: prepare_task.py Phase/Agent-Type Dispatch Mismatch

- **Bug.** Slash command bodies in `src/unit_25/stub.py` run `prepare_task.py --agent help` / `--agent hint` / `--agent redo` using the **phase name** form. But `prepare_task.py`'s dispatch at `src/unit_13/stub.py:1727` expects the **agent_type name** form (`help_agent`, `hint_agent`, `redo_agent`). When the user invokes `/svp:help` as a direct slash command, the orchestrator runs the body literally, dispatch falls through to the fallback at line 1748 (`"No specific dispatch for agent type: help"`), and the help agent receives an empty placeholder prompt instead of a structured task prompt. Affected commands: `/svp:help`, `/svp:hint`, `/svp:redo`. Unaffected: `/svp:ref` (phase == agent_type, lucky), `/svp:bug` and `/svp:oracle` (thin state-transition triggers post-S3-119 and S3-79; no longer call `prepare_task`).
- **Why it wasn't caught.** Same anti-pattern as S3-122 — the LLM orchestrator papers over the broken producer. When it receives the "No specific dispatch" fallback, it recognizes the help-agent role from the slash command name and the surrounding context, then answers from common sense rather than from a structured prompt. The command *appears* to work; only the raw Bash output reveals the fallback. The bug was invisible in all 4854 passing tests and only surfaced during manual empirical verification on 2026-04-14 when the user typed `/svp:help` in a fresh Claude Code session and read the literal command output.
- **Pre-existing since Pass 2 delivery.** `git show c75748d:src/unit_25/stub.py` confirms the body strings have been wrong since the initial delivery commit on 2026-03-30. Bugs S3-121, S3-122, S3-123 did not introduce or touch these lines.
- **Half-applied fix as the root cause.** `src/unit_13/stub.py` already had dual-form dispatch for two agents: `bug_triage`/`bug_triage_agent` at line 1737, and `coverage_review`/`coverage_review_agent` at line 1707. Someone previously encountered the phase/agent_type mismatch for those two, applied the correct fix (accept both forms), and stopped. Four other Group B agents — `help_agent`, `hint_agent`, `redo_agent`, `oracle_agent` — were left single-form. A half-applied fix is worse than no fix: it creates the false impression that the pattern is handled while leaving latent breakage for all the other cases in the same function.
- **Fix.** Extend dual-form dispatch uniformly to all Group B agents in `src/unit_13/stub.py`:
  - Line 1727: `elif agent_type == "help_agent":` → `elif agent_type in ("help", "help_agent"):`
  - Line 1729: `elif agent_type == "hint_agent":` → `elif agent_type in ("hint", "hint_agent"):`
  - Line 1735: `elif agent_type == "redo_agent":` → `elif agent_type in ("redo", "redo_agent"):`
  - Line 1743: `elif agent_type == "oracle_agent":` → `elif agent_type in ("oracle", "oracle_agent"):`
  Four lines changed. `reference_indexing` is left single-form because its phase name and agent_type name are identical — no drift possible. Slash command bodies in `src/unit_25/stub.py` are NOT modified; the fix is at the dispatcher, which matches the existing `bug_triage`/`coverage_review` precedent and avoids rewriting user-facing command documentation. Backward compatible — callers that pass the agent_type form still dispatch correctly.
- **Regression guards.** New file `tests/regressions/test_bug_s3_124_prepare_task_phase_dispatch.py` parameterizes over each Group B `(phase, agent_type)` pair and asserts that calling `prepare_task` with EITHER form produces a populated task prompt that does NOT contain the `"No specific dispatch for agent type"` fallback string. Also asserts that `reference_indexing` (the one single-form agent) still works with the phase-name form. Tests import from the derived `prepare_task` module per S3-103 stub-import discipline.
- **Lessons.**
  1. **When you find a dual-form dispatch, audit every dispatch in the same function.** Half-applied fixes are worse than no fix because they create the false impression that the pattern is handled. If a producer/consumer namespace mismatch required a dual-form match in one place, check every similar dispatch in the same function for the same risk. The trigger rule: whenever you edit a dispatch function to add `agent_type in ("X", "Y")` where `X` and `Y` are alternate names for the same concept, grep the entire function for other `agent_type == "..."` patterns and audit each one. If you cannot prove the namespace is uniform, add it to the dual-form dispatch.
  2. **Smart consumers paper over silent producers — part 4 of this pattern family.** S3-121 (commands/filenames), S3-122 (agents/frontmatter), S3-123 (plugin enablement/scope), and now S3-124 (prepare_task/dispatch): all four share the failure mode where an LLM orchestrator silently compensates for a producer-side bug. The bug is invisible until a human reads raw output. **Detection protocol going forward**: for any producer→consumer contract where the consumer is the LLM orchestrator, assume the LLM will paper over minor drift. Add explicit tests that exercise the contract without involving the LLM — parse outputs for literal fallback strings, inspect files for expected keys, assert registered identifiers match source-of-truth identifiers. The orchestrator is a production consumer, not a test harness.
  3. **Empirical end-to-end verification surfaces bugs that static test suites cannot.** S3-124 was invisible in all 4854 passing tests. It surfaced only when the user typed `/svp:help` in a real Claude Code session and looked at the literal Bash output of the `prepare_task.py --agent help` call. The static tests exercised the code paths at the wrong layer — they verified `prepare_task` internals work correctly when called with `agent_type="help_agent"` (which they do), but never verified that the slash command body's literal invocation (`--agent help`, phase form) produces a non-fallback result. **Going forward**: every user-facing surface (slash commands, CLI entry points, gate prompts) should have an end-to-end test that runs the exact literal command the user would type and asserts on the raw output. This is test-at-the-surface, not test-at-the-function.
- **Pattern.** P28 (NEW — Half-Applied Dispatch Fix). When a producer/consumer namespace mismatch is partially fixed by adding a dual-form dispatch to one or two cases but not others, the remaining single-form cases are latent bugs that can silently mask for years. Sibling of P26 (Generator-Side Identifier Transformation) and P27 (Default Scope Leak) — all three concern silent drift between what the code says and what the system actually does at runtime. Mitigation checklist: (a) grep for every dispatch with a similar structure after fixing any one of them, (b) prefer dual-form dispatches as the default for producer/consumer contracts with distinct namespaces, (c) every dispatch site should have a regression test that exercises both forms.
- **Cross-reference.** Fourth member of the Claude Code platform-contract drift family (S3-121, S3-122, S3-123, S3-124). Also a sibling of the half-fixed pattern family: the previous half-fixes for `bug_triage`/`coverage_review` in `src/unit_13/stub.py` show that SVP has seen this exact anti-pattern before. The dispatch function should be audited in its entirety after this fix to ensure NO other single-form dispatches have a sibling phase-name form that could drift. Related audit candidates: `dispatch_agent_status` in routing (uses `agent_type` dispatch), `dispatch_command_status` (uses command-name dispatch), any other function that takes a name and dispatches. **S3-125 (next entry) addresses one of those audit candidates — `dispatch_command_status` had a separate half-applied rename from S3-121.**

## S3-125: `svp_bug_entry` Orphan Prefix in `dispatch_command_status`

- **Bug.** `src/unit_14/stub.py` `dispatch_command_status` had 14 `command_type` dispatch branches. Thirteen were bare-named (`stub_generation`, `test_execution`, `quality_gate`, `unit_completion`, `compliance_scan`, `structural_check`, `lessons_learned`, `debug_commit`, `stage3_reentry`, `oracle_start`, `oracle_test_project_selection`, `oracle_gate_7a`, `oracle_gate_7b`). One — `svp_bug_entry` — carried an `svp_` prefix. A 13:1 ratio against the outlier. No other dispatcher shares the namespace; the prefix added zero disambiguation value inside a file whose entire purpose is SVP state dispatch.
- **Asymmetry explicitly noted in existing tests but never treated as a bug.** `tests/unit_25/test_unit_25.py:28` docstring verbatim: *"bug uses --command svp_bug_entry, oracle uses --command oracle_start"*. The drift was observed, documented, and accepted as if it were a feature. This is the exact shape of a latent bug hiding in plain sight: someone saw the inconsistency, wrote it down, and moved on instead of fixing it. **Lesson: if you find yourself documenting an asymmetry as if it were a feature, stop and check whether it's actually a bug. Asymmetries in a dispatch table are almost always bugs.**
- **Historical trace — half-applied rename from S3-121.** `svp_bug_entry` was introduced by Bug S3-119 to let `/svp:bug` bootstrap a debug session from pipeline_complete. At that time, the slash command file was `svp_bug.md` (with prefix), so the new dispatch key was named symmetrically: `svp_bug_entry`. S3-121 later renamed `svp_bug.md` → `bug.md` but explicitly scoped out state-machine command names as "not filenames". That conservative scoping was wrong: `svp_bug_entry` was the ONLY `svp_`-prefixed state-machine command, so leaving it alone created the 13:1 asymmetry. The S3-121 plan said *"state-machine commands like `svp_bug_entry` and `oracle_start` are untouched"* — naming two commands implies a class of multiple, but there was only one.
- **Fix.** Hard rename `svp_bug_entry` → `bug_entry` everywhere. No dual-form dispatch, no backward compatibility layer. No external callers exist — every reference is in files under SVP's control. The new name preserves the `_entry` suffix because the spec §12.18.13 transition is `null → "triage"` — an *entry* into an existing state machine, semantically distinct from `oracle_start` (which creates a new nested session workspace). Making the two symmetric by renaming to `bug_start` would flatten a meaningful semantic distinction.
- **Regression guards.** New `tests/regressions/test_bug_s3_125_bug_entry_rename.py` with three tests: (1) positive — `dispatch_command_status(state, "bug_entry", "")` creates debug session correctly; (2) **negative sentinel** — `dispatch_command_status(state, "svp_bug_entry", "")` does NOT create a debug session, guarding against a future re-introduction of the old name; (3) body-text check — `COMMAND_DEFINITIONS["bug"]` contains `--command bug_entry` and does NOT contain `--command svp_bug_entry`. The negative sentinel is the load-bearing guard: a future refactor that silently re-introduces `svp_bug_entry` (e.g., by copy-paste from a historical commit) would fire this test immediately.
- **Historical references NOT modified.** The S3-119 catalog entry (spec §24.132) and the S3-121 catalog entry (spec §24.134) reference the original `svp_bug_entry` name in their historical prose describing what was done at the time. These are frozen historical records, not live references. They are preserved unchanged. Only live references (dispatch key, slash command body, blueprint contract bullets, live test assertions) are renamed. **Lesson: when renaming anything, classify every reference as "live" or "historical" before editing. Live references describe the current state of the code; historical references describe what was done at a point in time and must not be rewritten retroactively or they become lies.**
- **Test filename NOT renamed.** `tests/regressions/test_bug_s3_119_svp_bug_bootstrap.py` retains its filename. The file tests Bug S3-119's behavior (bootstrap from pipeline_complete), which is still valid under the renamed dispatch key. Renaming the file would decouple it from its bug number and confuse future auditors trying to trace a specific bug to its regression test. **Lesson: test filenames bind to bug numbers, not to command names. Rename command names freely; never rename a bug regression test file.**
- **Lessons.**
  1. **Asymmetries in a dispatch table are almost always bugs.** If a dispatch function has N handlers following convention X and M handlers following convention Y, and N >> M, the M handlers are the outlier. Either they need an explicit exception in the spec with a documented reason, or they are a bug waiting to be cleaned up. Never accept an asymmetry as "just how things are" without interrogating it.
  2. **Conservative scoping on cross-layer renames must be accompanied by an audit of the excluded layer.** S3-121 excluded state-machine command names from its scope as a surgical decision, but didn't audit whether the excluded layer had the same drift pattern. It did — on a single outlier. An audit would have taken five minutes and prevented this bug from living for another month. **Going forward**: when scoping out a layer from a rename, grep that layer for the old prefix, and if the layer contains MORE than zero occurrences, include it or justify the exclusion with a specific reason.
  3. **Cosmetic bugs are still bugs.** S3-125 is not functionally broken — `svp_bug_entry` dispatches correctly. But it's a latent cognitive trap: readers of `dispatch_command_status` will see the outlier and assume the `svp_` prefix is load-bearing (it isn't), which creates a false mental model that compounds over time. Fixing cosmetic drift when it's cheap prevents it from becoming structural drift later.
  4. **"If you find yourself documenting an asymmetry as if it were a feature, it's a bug."** The `tests/unit_25/test_unit_25.py:28` docstring is a perfect example of this failure mode: someone looked at the mismatch between `svp_bug_entry` and `oracle_start`, felt the need to explain it, and wrote documentation instead of raising a bug. Documentation that explains "why these are different" is sometimes warranted; documentation that explains "this one is different, we just accept it" is a code smell. Treat such docstrings as bug reports the author forgot to file.
- **Pattern.** P28 (Half-Applied Dispatch Fix), extended to the state-machine command namespace. Fourth member of the bug family: S3-121 (command files), S3-122 (agent frontmatter), S3-124 (phase/agent_type dispatch), S3-125 (state-machine dispatch key).
- **Cross-reference.** Fifth consecutive platform-contract drift bug surfaced during manual verification in the S3-121→S3-125 sweep. S3-121 was triggered by a debrief cross-project advisory; S3-122 was caught by the S3-121 triage audit (sibling-surface check); S3-123 was another debrief advisory; S3-124 was caught by empirical `/svp:help` testing during S3-123 verification; S3-125 was caught when I asked the user what they meant by "address /svp:bug and /svp:oracle" and discovered the 13:1 asymmetry while investigating. **The bug family is evidently still open — each fix surfaces the next one.** Next audit candidates: `dispatch_agent_status` in routing (may have its own single-vs-dual-form inconsistency), the remaining non-dispatch surfaces that use `svp_` prefixes anywhere (hooks, environment variables, log markers), and — most importantly — the **full scope of the `svp_` prefix convention across the codebase**. A proactive sweep for `svp_` as a producer-side prefix should be done before the next manual verification cycle.

## S3-126: Universal Manual Bug-Fixing Protocol in Tier 1 CLAUDE.md

- **Bug (feature gap).** The CLAUDE.md Tier 1/Tier 2 split introduced by Bug S3-99 placed the Manual Bug-Fixing Protocol (Break-Glass Mode) exclusively in `CLAUDE_MD_SVP_ADDENDUM` (Tier 2), appended only for E/F self-build projects. For A-D archetypes (`python_project`, `r_project`, `claude_code_plugin`, `mixed`), no break-glass protocol was ever installed. When SVP routing was too broken to function in an A-D project, the main session had no written fallback procedure and had to improvise from general instincts. Additionally, neither tier prescribed a final **commit to git** step — fixes could be applied and verified but the protocol never closed the loop by recording the change in version control, leaving repositories in an inconsistent "fixed on disk, uncommitted" state after manual intervention.
- **Why S3-99 missed this.** S3-99 framed the protocol as an "SVP self-build" concern because it was the SVP authors who most frequently found themselves in break-glass mode. The mental model was "break-glass is for when we're debugging SVP itself", so the protocol went into the SVP-only addendum. The omission only surfaced when the user — as the project lead on an in-flight Pass 2 — asked for a feature that would install the protocol in all archetypes, pointing out that A-D projects also need a fallback procedure and the commit step had always been missing even from Tier 2. **Lesson: when you split documentation into "universal" and "specialized" tiers, interrogate the universal tier for completeness against every downstream consumer, not just the one you authored the split for. S3-99's Tier 1 was minimally viable for routing-only use; it was never pressure-tested against the break-glass use case.**
- **Scope mismatch between tiers.** The original Tier 2 addendum was SVP-self-build-specific: references to `src/unit_*/stub.py` (stubs as source of truth), `sync_workspace.sh` (workspace/repo sync), test-from-both-workspace-and-repo, and deployed plugin artifacts under `svp/commands/`, `svp/skills/`, `svp/agents/`, `svp/hooks/`. These concepts are meaningless for A-D projects, which have a single source tree and a simple delivered repo. Simply copy-pasting Tier 2 into Tier 1 was therefore not viable — the universal protocol had to be generic, with E/F-specific concerns layered on top as an *override addendum* rather than as a standalone replacement. **Lesson: when a "universal" tier and a "specialized" tier end up with a shallow overlap (like "both have a bug-fixing protocol") but distinct content, the refactor is to make the specialized tier an *override* of the universal tier, not a parallel restatement. Parallel restatements drift.**
- **Workspace/repo awareness lives in pipeline state, not in the template.** All SVP projects end up with a delivered repo at `{project_root.parent}/{project_name}-repo/` (a sibling directory of the workspace). The workspace is NOT itself a git repository — git lives only in the sibling. The delivered repo's absolute path is persisted in `.svp/pipeline_state.json::delivered_repo_path` after Stage 5 succeeds (Bug S3-112). The break-glass protocol in CLAUDE.md tells the orchestrator to read `delivered_repo_path` from state at commit time, navigate to the sibling directory, and commit there — never in the workspace. **Lesson: when multiple directories matter (workspace vs. delivered repo), the template cannot hardcode a path because the path depends on the project name. The pipeline state is the right source of truth, and the template references it by name rather than substituting.**
- **Commit message format from profile, not template.** `project_profile.json::vcs.commit_style` holds the format choice (`"conventional"`, `"freeform"`, or `"custom"`) captured at Stage 1, and `vcs.commit_template` holds the template string for `"custom"`. The break-glass protocol instructs the orchestrator to read the profile at commit time — the template cannot bake the style in because CLAUDE.md is created at Stage 0 (by `create_new_project`) before the profile exists. **Lesson: any instruction in CLAUDE.md that depends on a profile field must direct the orchestrator to read the profile dynamically at the moment of use, not substitute at render time. Render-time substitution is impossible for profile-dependent instructions because the profile doesn't exist yet when the template renders.**
- **Fix.** Three coordinated changes in `src/unit_29/stub.py`:
  1. `CLAUDE_MD_TEMPLATE` rewritten to include a universal `## Manual Bug-Fixing Protocol (Break-Glass Mode)` section with an 8-step cycle (DIAGNOSE → PLAN → EXECUTE → EVALUATE → LESSONS LEARNED → REGRESSION TESTS → VERIFY → COMMIT TO GIT). The COMMIT TO GIT step includes a "Workspace vs. Delivered Repo" subsection that explicitly warns the orchestrator that the workspace is NOT a git repository and that git lives in the sibling `{project_name}-repo/` directory, with the substitution happening at render time so each project sees its actual repo name.
  2. `CLAUDE_MD_SVP_ADDENDUM` rewritten as an override addendum under the top-level heading `## SVP Self-Build Override`. It does NOT restate the cycle — it specifies overrides (EXECUTE edits stubs, VERIFY from both directories) and extra steps (DEPLOYED ARTIFACTS, SYNC) that layer on top of Tier 1 before the terminal COMMIT TO GIT step.
  3. `enrich_claude_md_for_svp_build` idempotency marker changed from `"Manual Bug-Fixing Protocol"` (now present in Tier 1 by default) to `"SVP Self-Build Override"` (the Tier 2 heading, unique to Tier 2). Without this change, the old short-circuit would fire on every E/F project and Tier 2 would never append. **Lesson: when you promote content from a specialized tier to a universal tier, the old idempotency marker becomes a footgun — it will short-circuit every call because it's now present everywhere. Always re-derive the marker from text that is unique to the tier being checked.**
- **Regression guards.** New `tests/regressions/test_bug_s3_126_universal_break_glass.py` with 23 tests across four classes: (1) `TestTier1UniversalProtocol` locks every invariant of the universal cycle — RULE 0, ordered step sequence, commit-to-git step, sibling-repo substitution, references to `delivered_repo_path` / `project_profile.json` / `vcs.commit_style`, default `fix: <bug-id> <short-desc>` format, and absence of Tier 2 overrides; (2) `TestTier2OverrideAddendum` locks the Tier 2 marker heading, self-build content (stubs, sync, deployed artifacts, test-from-both), and the negative invariant that Tier 2 does NOT duplicate the full COMMIT TO GIT step block from Tier 1; (3) `TestEnrichClaudeMdIdempotency` tests the new marker on three scenarios — plain Tier 1 (should append Tier 2), double-call (should be byte-equal), and missing CLAUDE.md (should not raise); (4) `TestCreateNewProjectShipsTier1` verifies that a fresh A-D `create_new_project` writes CLAUDE.md with Tier 1's universal protocol installed from Stage 0 and with Tier 2 absent until enrichment runs. Additionally, the pre-existing Bug S3-99 regression tests were updated to reflect the new invariants: `test_tier1_no_bug_protocol` → `test_tier1_has_universal_bug_protocol`, `test_claude_md_tier1_only` updated to check for the Tier 2 marker's absence rather than the Tier 1 protocol's absence, `test_idempotent` updated to count the new marker, and `test_ad_project_clean` flipped from "protocol absent" to "protocol present, but Tier 2 overrides absent".
- **Deployed artifact propagation.** Unit 23's `assemble_svp_workspace_artifacts` imports `CLAUDE_MD_TEMPLATE` and `CLAUDE_MD_SVP_ADDENDUM` from Unit 29 and writes `CLAUDE.md = TEMPLATE + ADDENDUM` to the delivered E/F repo root. This was intentionally NOT modified: because it imports the constants, my Tier 1/Tier 2 edits propagate automatically into the E/F delivered repo at next Stage 5. A-D delivered repos do not receive a CLAUDE.md at all (`assemble_svp_workspace_artifacts` only runs when `is_svp_build` is true), which is correct — A-D delivered repos are clean product repositories and should not carry SVP orchestration metadata. **Lesson: when a template constant is imported by multiple writers (Unit 29's `create_new_project` for Stage 0, Unit 23's `assemble_svp_workspace_artifacts` for Stage 5), editing the constant once propagates to all writers. Don't duplicate the template — centralize it and let all writers import.**
- **Lessons.**
  1. **Universal tiers must be audited against every downstream consumer, not just the author's primary use case.** S3-99 designed Tier 1 around the SVP authors' routing-only needs, then carved out bug-fixing into Tier 2 as "SVP self-build only". The carve-out was correct for SVP-self-build-specific content (stubs, sync), but wrong for the universal *existence* of a break-glass protocol. Every downstream archetype was a silent consumer that needed the protocol but never got it. **Going forward**: when splitting a document into universal and specialized tiers, list every downstream consumer and verify the universal tier is complete for each one. If a feature is needed by all consumers, it goes in the universal tier — even if the author's primary consumer happens to be the specialized one.
  2. **Override tiers layer on top of universal tiers; they do not restate them.** Pre-S3-126 Tier 2 was a standalone break-glass protocol that happened to be SVP-specific. Post-S3-126 Tier 2 is an override addendum that references Tier 1's step names and specifies "for the EXECUTE step, do X instead; add a SYNC step before COMMIT TO GIT". The override form is more maintainable: when Tier 1's cycle changes, Tier 2 automatically inherits the change for any step it doesn't override. Parallel restatements drift — override forms don't.
  3. **Idempotency markers must be tier-unique, not content-unique.** Pre-S3-126, the marker `"Manual Bug-Fixing Protocol"` was sufficient because that string appeared only in Tier 2. Post-S3-126, it appears in Tier 1 too, so it's useless as a Tier 2 marker. The new marker `"SVP Self-Build Override"` is the Tier 2 top-level heading and exists nowhere else in Tier 1. **Going forward**: when promoting content across tiers, audit every idempotency check that references the promoted content — old markers become silent footguns that short-circuit without appearing broken.
  4. **Profile-dependent instructions must be dynamic references, not static substitutions.** The CLAUDE.md template is rendered at Stage 0 by `create_new_project`, but `project_profile.json` is authored at Stage 1. Any template that depends on profile fields must therefore direct the orchestrator to read the profile at runtime, not substitute the profile value into the rendered template. This is a general principle for any template rendered early in a lifecycle: it cannot bake in values from later stages.
  5. **"All projects end up with a repo" is a load-bearing invariant that the break-glass protocol relies on.** Before S3-126 I initially assumed only E/F projects had a delivered repo and was about to gate the commit step on an `is_svp_build` check. The user corrected me: **every** SVP archetype (A-F) produces a delivered sibling repo via Stage 5. This is why the commit step can be universal — because the delivery target is universal. **Going forward**: when designing a cross-archetype feature, explicitly verify the load-bearing invariants against every archetype before specializing for one. The urge to specialize is usually premature when the underlying invariant is already universal.
- **Pattern.** P7 (Specification Omission) in the universal-tier category. S3-99 documented the split but framed the break-glass protocol as an SVP-specific concern, leaving A-D archetypes uncovered. The omission surfaced only when the user observed that A-D projects also need a fallback and a closing commit. This is a new pattern variant worth naming: **P29 — Universal-Tier Undercoverage**. A tiered document split where the "universal" tier omits content that every downstream consumer needs, because the split was authored from the perspective of the specialized consumer. Detection: audit universal-tier contents against every archetype at design time, not at report time.
- **Cross-reference.** Scoped through break-glass mode (Manual Bug-Fixing Protocol) during Pass 2 bug-fixing, triggered by the user request to generalize the protocol to all archetypes and add a commit-to-git step. Distinct from Section 43.9's "orchestrator break-glass protocol" (which handles in-pipeline exhaustion during E/F self-builds) — S3-126 is about the CLAUDE.md Manual Bug-Fixing Protocol that governs direct manual intervention when SVP routing itself is broken, not about pipeline retry exhaustion.

## S3-127: `ensure_project_settings()` Wrote a Cache-Parent Path That Has No `marketplace.json`

- **Bug.** After the S3-123 migration to project-scope plugin activation, launching Claude Code from an SVP pipeline directory crashed at plugin load time: `Marketplace file not found at /Users/<user>/.claude/plugins/cache/svp/svp/.claude-plugin/marketplace.json`. The path being dereferenced does not exist and cannot exist — `~/.claude/plugins/cache/<marketplace>/<plugin>/` is Claude Code's internal plugin cache layout, **not** a marketplace root. The only real `marketplace.json` on the machine was in the source repo at `<svp-source-repo>/.claude-plugin/marketplace.json`. SVP was silently inaccessible from every project-scoped launch despite the S3-123 fix having "landed" and its regression tests passing.
- **Root cause chain.** `svp_launcher.main()` called `_find_plugin_root()`, which iterated standard locations in order. The second location was `~/.claude/plugins/cache/svp/svp`, containing a leftover snapshot from a previous user-scope install (a `2.2.0/` version subdir with a valid `plugin.json`). `_find_plugin_root()` happily returned `~/.claude/plugins/cache/svp/svp/2.2.0/`. `ensure_project_settings()` then computed `marketplace_root = plugin_root.parent.resolve()` → `~/.claude/plugins/cache/svp/svp/` and wrote that string into `extraKnownMarketplaces.svp.source.path`. The invariant "plugin_root.parent contains marketplace.json" holds for **source-repo layouts** (`<repo>/svp/` whose parent `<repo>/` holds the marketplace file) but is **false for cache layouts**, where the parent is just an intermediate directory of Claude Code's internal cache (no marketplace.json ever lives there).
- **Why the S3-123 test suite didn't catch it.** `tests/regressions/test_bug_s3_123_project_settings.py` fixture `plugin_root` constructed a synthetic `marketplace_root / "svp"` layout but **never wrote a real `marketplace.json` file** inside `marketplace_root / ".claude-plugin/"`. The test asserted structural correctness ("the parent is written correctly") rather than semantic correctness ("the written path actually resolves to a valid marketplace root"). The cache-layout failure mode was therefore invisible to the suite — the fixture was too forgiving. **Lesson: a fixture that builds a fake marketplace must create every file the marketplace load path expects. Anything less is an invariant the test is no longer enforcing.**
- **Why this didn't manifest during S3-123's own verification.** When S3-123 was developed, the user still had SVP registered user-scope at `~/.claude/settings.json` with `extraKnownMarketplaces.svp.source.path = <svp-source-repo>`. Claude Code was populating the cache from a real marketplace root. After the S3-123 user-side migration (`claude plugin uninstall svp@svp --scope user` and `"extraKnownMarketplaces": {}` in user-scope settings), the user-scope pointer to the real marketplace was **removed**. From that point forward, every `svp resume` rewrote the project's `.claude/settings.json` with the stale cache-parent path — a self-healing mechanism actively **corrupting** its target. **Lesson: a self-heal that rewrites a stored value must validate the new value against the same invariant it enforces for fresh writes. A "widen on rewrite" self-heal is strictly worse than no self-heal, because it replaces a known-stale path with a freshly-computed invalid one.**
- **Section 4.4 already documented the invariant.** The S3-123 common-mistakes checklist in spec §4.4 states: *"Writing the settings file before the marketplace directory actually exists on disk — the path in `extraKnownMarketplaces.svp.source.path` must resolve to a real directory with `.claude-plugin/marketplace.json` inside."* The checklist was correct. `ensure_project_settings()` did not enforce it. Documentation without enforcement is a permission slip. **Lesson: every bullet in a "common mistakes" checklist must be convertible to an assertion or a validation call in the code that the checklist is about. If it cannot be, promote the item to an architectural invariant that the test suite enforces structurally. A checklist is a seed for test cases, not a substitute for them.**
- **Fix.** Four coordinated changes in `src/unit_29/stub.py`:
  1. **New helper `_find_marketplace_root(plugin_root)`** with a three-step resolution order: (a) `SVP_MARKETPLACE_ROOT` env var if set and valid, (b) `plugin_root.parent` if it is a valid marketplace directory, (c) walk up from `Path(__file__).resolve()` looking for the nearest ancestor that is a valid marketplace root. Returns `None` if none reachable. The `__file__` walk-up is the reliable fallback: the launcher is itself part of the plugin, so its own location on disk transitively identifies a valid marketplace when imported from a source tree.
  2. **New helper `_is_valid_marketplace_dir(path)`** encodes the validation rule in one place: the directory is valid iff `.claude-plugin/marketplace.json` exists, is parseable JSON, has a `plugins` array, and at least one entry has `name == "svp"`. All three conditions must hold. Existence of the file alone is not sufficient.
  3. **Two-pass `_find_plugin_root()`.** First pass returns only candidates whose parent is a valid marketplace root. Second pass falls back to the pre-S3-127 "any valid plugin.json" behavior. The effect is that a user with both a source-repo install and a stale cache snapshot gets the source repo. A user with only the cache still falls back to it, and `ensure_project_settings()` then hard-fails with a clear message rather than silently writing a broken path.
  4. **Hard-fail `ensure_project_settings()`.** The helper now calls `_find_marketplace_root(plugin_root)`. If the result is `None`, it raises `FileNotFoundError` with an actionable message instructing the user to set `SVP_MARKETPLACE_ROOT`. The self-heal clause is widened: it now rewrites `extra["svp"]` whenever the stored path does not contain `.claude-plugin/marketplace.json`, not only when it differs from the freshly-computed one. This converts S3-127 from silent corruption into a loud failure on the next launcher invocation.
- **Regression guards.** New `tests/regressions/test_bug_s3_127_marketplace_path.py` with five classes covering: (1) hard-fail when no marketplace is reachable and the `__file__` walk-up is isolated from any marketplace ancestry; (2) the exact cache layout (`cache/<m>/<p>/<version>/`) is rejected; (3) `SVP_MARKETPLACE_ROOT` env var takes precedence over `plugin_root.parent`, and an invalid env var value raises immediately; (4) a pre-existing stale cache-parent path in settings.json is self-healed on the next run; (5) `_find_plugin_root()` prefers a source-repo candidate over a cache-layout candidate when both are present, falling back to the cache only if no source repo is discoverable. Plus an extra `TestIsValidMarketplaceDir` class that locks the three-part invariant (file exists, plugins array, svp entry) against edge cases. Additionally, the pre-existing S3-123 fixture was updated to write a real `marketplace.json` inside the fake marketplace root so that S3-123's tests continue to exercise the fast path in `_find_marketplace_root` — closing the gap that let S3-127 slip through in the first place.
- **Lessons.**
  1. **"Parent of plugin dir" and "marketplace root" are two different things.** In the source-repo layout they happen to coincide — a convenient accident — but in Claude Code's plugin cache layout they diverge. Any code that computes a path by walking *up* from a discovered location must validate the result against the **invariant it actually needs** (here: `marketplace.json must exist`), not the typical layout the author had in mind. The shape of the invariant is the test, not the shape of the typical case.
  2. **Silent corruption is worse than loud failure.** S3-123's original self-heal rewrote stale paths to the (incorrectly computed) parent of the discovered plugin. Once the user migrated off user-scope, every `svp resume` overwrote the project's settings file with a broken path — the helper was the vector for the corruption, not just a bystander to it. A loud `FileNotFoundError` on the next run would have surfaced the bug in minutes. Silent path rewriting let it fester for a full release cycle. **Going forward**: a self-heal that rewrites a stored value must validate the replacement against the same invariant that would apply to a fresh write. "Replace stale with fresh" is not an invariant — "replace invalid with valid, or refuse to write" is.
  3. **Fixture realism is a test invariant.** The S3-123 fixture used `tmp_path / "svp_marketplace_repo" / "svp"` as a fake plugin root but never created a real `marketplace.json` file. The tests passed because the pre-S3-127 helper didn't check for one — but the fixture encoded a weaker invariant than the production code's callers. Any test fixture that represents a production artifact must create the same files the production artifact contains, even if the code under test doesn't currently read them. A fixture that is more permissive than reality is a fixture that hides bugs. **Mitigation protocol**: when designing a fixture, list the files the real artifact would contain, and create every file whose absence would be invisible to the current code but detectable to a future version. Err on the side of building more.
  4. **Documented invariants must be enforced in code.** Spec §4.4's common-mistakes checklist listed "the marketplace path must resolve to a real directory with `.claude-plugin/marketplace.json`" as a mistake to avoid. Humans honored it during review. The helper did not. A year later, a leftover cache snapshot turned the warning into a bug. **Going forward**: every bullet in a "common mistakes" checklist must be convertible to an assertion or a validation call in the code that the checklist is about. If the bullet is not enforceable by the code itself (e.g., it describes a human-only workflow mistake), move it out of the checklist and into a README / operational runbook. A checklist embedded in a spec is a seed for test cases, not a substitute for them.
  5. **The launcher's own `__file__` is the most reliable plugin location discovery mechanism.** Every previous plugin discovery strategy searched external directories — environment variables, Claude Code's cache, XDG config paths, system prefixes. All of those are vulnerable to the user having multiple installs, stale installs, or partial installs. But `Path(__file__).resolve()` on the launcher module itself is, by construction, the exact location where the launcher code was loaded from. Walking up from it finds the marketplace root transitively — as long as the launcher was imported from a source tree rather than a site-packages reinstall. **Going forward**: when a plugin needs to find itself at runtime, prefer `__file__`-based discovery over directory searches. `__file__` is always right for "where did my code come from"; directory searches guess.
- **Pattern.** **P30 (NEW — Invariant Documented But Not Enforced)**. A spec or documentation explicitly warns about a mistake, but the code that would prevent the mistake does not encode the check. Human reviewers honor the warning during design and review; later, a change in circumstances (a stale cache, a moved repo, a user migration) violates the warning and the code silently propagates the violation. Sibling of P28 (Half-Applied Dispatch Fix) and P27 (Default Scope Leak): all three are instances of "the code did half the job and the documentation carried the rest" — and all three eventually blew up when the documentation stopped being consulted in the moment.
- **Cross-reference.** Sixth consecutive platform-drift bug in the S3-121 through S3-127 family. Discovered during a user-reported launch failure rather than during automated testing. Each bug in this family has surfaced a different angle on the same underlying issue: the gap between "what SVP ships" and "what Claude Code actually sees at runtime." S3-121 and S3-122 concerned filename drift. S3-123 concerned scope leak. S3-124 and S3-125 concerned dispatch-namespace drift. S3-126 concerned a universal-tier omission. S3-127 concerns a documented invariant that was never encoded. **The common thread: SVP's platform-contract layer is under-enforced. The next audit candidate is a comprehensive sweep of every assumption SVP makes about Claude Code's filesystem layout, cache structure, plugin discovery, and settings file precedence — encoded as runtime assertions rather than review-time documentation.**

## S3-128: Preflight Detection of Pre-S3-123 User-Scope Plugin Leak

- **Bug (feature gap).** Bug S3-123 migrated SVP to project-scope plugin activation and documented the user-side migration (run `claude plugin uninstall svp@svp --scope user`, then run the launcher in each pipeline directory). The migration is opt-in. Users who upgraded SVP without performing it retained `enabledPlugins["svp@svp"] = true` in `~/.claude/settings.json` indefinitely, leaking SVP into every Claude Code session on the machine regardless of cwd. The launcher never detected whether the migration was actually performed — the fix's own code runs correctly on freshly-migrated systems, but unmigrated systems were invisible to the runtime. **Lesson: an opt-in user migration without a runtime detection function is a silent-residual-state failure. The fix works, the documentation is correct, but the signal that tells a user "you didn't finish the migration" does not exist.**
- **How this bug came out of an audit, not a crash.** Unlike every other S3-12x bug, nobody was bitten by S3-128 before it was fixed. It surfaced during the post-S3-127 runtime enforcement sweep — a proactive audit of every documented-but-unenforced invariant in the S3-121 through S3-127 family. The audit classified each invariant as (a) already covered by an existing regression test, (b) already enforced at runtime, or (c) unenforced. The first two buckets accounted for every filename/frontmatter/dispatch invariant in the family. The third bucket contained exactly one item: the opt-in migration step from S3-123. That's what S3-128 is.
- **The audit's real output is what it retired, not what it added.** The sweep was initially scoped to four MUST-tier items — I1 (command filename structural check), I2+I8 (agent frontmatter structural check), I6 (this item, user-scope leak detection), and I11 (deployed artifact staleness check). A dogfood pre-check (running the planned invariants against the real `svp2.2-pass2-repo/svp/` before writing any code) surfaced that I2+I8's planned invariant was **inverted** — the planned rule (`name == filename_stem.replace("_", "-")`) was the pre-S3-122 bug, and the correct post-S3-122 rule (`name == filename_stem` verbatim) was already enforced by `tests/regressions/test_bug_s3_122_agent_name_drift.py`. I1 was similarly covered by `tests/regressions/test_bug_s3_121_command_double_prefix.py` with seven tests including positive regex checks, negative sentinels, and body-heading consistency. I11 required auditing Unit 23's assembly flow and was deferred to a standalone bug entry (S3-129, pending). The sweep collapsed from four items to one. **Lesson: before adding a runtime assertion for a "documented invariant", grep the existing regression test corpus for the invariant's signature. If there's already a test enforcing it, the runtime assertion is duplication, and duplication creates drift between the two enforcement sites — which is itself a new bug class.**
- **Lesson: the dogfood pre-check saved the sweep.** Running the planned invariants against the real repo *before* writing any enforcement code caught the inverted rule in under a minute. If I had written the structural check first and the dogfood check second, the check would have failed on 21 of 21 agent files, and I'd have had to unwind the work and re-examine the premise. The dogfood step costs five minutes and prevents a class of bugs where the check is technically correct but enforces the wrong rule. **Mitigation protocol**: whenever you plan a structural check that compares source-of-truth values against a derived form, run the check against the real repo before committing to the derived-form formula. If the check fails on live data, the formula is wrong. If it passes, proceed.
- **Fix (Bug S3-128).** Three coordinated changes in `src/unit_29/stub.py`:
  1. **New helper `check_user_scope_svp_leak() -> Optional[str]`.** Reads `~/.claude/settings.json` defensively and returns an advisory message when `enabledPlugins["svp@svp"] is True`, or `None` otherwise. Every failure mode (missing file, corrupt JSON, non-dict root, non-dict `enabledPlugins`, `svp@svp` missing or set to `False`/`0`/`"true"`/`None`) is treated as "no leak detected" and returns `None`. The helper is pure detection — read-only, non-raising, side-effect-free. The advisory message when non-`None` contains the substrings `svp@svp` and `user scope`, references the migration command `claude plugin uninstall svp@svp --scope user`, and cites spec §4.4.
  2. **Preflight wire-in.** `preflight_check()` gains a new advisory check #2a, called right after the existing "SVP plugin loaded" check. If `check_user_scope_svp_leak()` returns a non-`None` message, preflight prints it to stdout with a `!` advisory marker (matching the existing API-credentials pattern at check #3). The check does NOT append to the errors list. Preflight still passes even when the advisory fires.
  3. **Regression guards.** New `tests/regressions/test_bug_s3_128_user_scope_leak_detection.py` with 26 tests across four classes. `TestReturnsNoneWhenNoLeak` covers nine non-leak paths including `svp@svp` set to `False`/`1`/`"true"`/missing. `TestReturnsNoneOnCorruptInput` covers six corrupt-input paths including invalid JSON, empty file, list/string root, list/string `enabledPlugins` — the helper must never raise. `TestReturnsAdvisoryWhenLeaked` locks the four shape invariants on the advisory string (plugin identifier, "user scope" phrase, migration command, §4.4 citation). `TestPreflightIntegration` verifies the advisory is printed to stdout with the `!` marker when the leak is present, is silent when absent, does NOT affect the errors list, and does NOT crash on corrupt user settings. All tests use `monkeypatch.setattr(Path, "home", lambda: tmp_path)` to redirect `Path.home()` so the suite is hermetic on every developer's machine regardless of their own user-scope settings.
- **Why WARN, not FAIL.** Preflight is hit on every launcher invocation. Failing preflight blocks `svp resume` entirely. A user who legitimately wants dual-scope enablement during a grace period (e.g., testing the migration on one project while other projects still rely on user scope) would be locked out of SVP until they performed the migration. Advisory mode lets such users ignore the warning while still surfacing the residual state to users who want to know. **Lesson: when converting documentation to runtime enforcement, default to advisory first and upgrade to blocking only if the advisory is demonstrably ignored. Blocking is a one-way door; advisory is a two-way door.**
- **Lessons.**
  1. **Opt-in migrations need residual-state detection functions.** Every bug-fix spec entry that documents a user-side migration step (`run X command`, `delete Y file`, `edit Z setting`) must be accompanied by a runtime detection function in the same or adjacent commit. The detection function does not need to *fix* the residual state — it just needs to *report* it. Silent residual state is the failure mode; loud detection is the fix. **Going forward**: every bug entry with a "User-side migration" section must have a corresponding "Runtime detection" section in the same spec entry, or an explicit sentence explaining why detection is not possible.
  2. **The dogfood pre-check is a first-class step in runtime-assertion design.** When the invariant is going to be enforced against real repo artifacts, running the planned invariant against the real artifacts BEFORE writing enforcement code catches inverted or outdated invariant formulas in minutes. The cost is trivial; the value is saved re-work and prevented regressions of previously-fixed bugs. **Going forward**: any structural check that scans `svp/<asset>/` must include a dogfood-pre-check step in the plan phase — not as a suggestion but as a gate. The dogfood check saved the S3-128 sweep from shipping an inverted I2+I8 rule that would have re-introduced S3-122 under the guise of fixing it.
  3. **Audits usually retire more work than they add.** The S3-128 sweep audited four candidate items and promoted one to implementation. The three retired items were all already enforced — two by existing regression tests, one (I11) deferred because it needed more design work. The sweep's real output is the realization that the existing regression-test corpus is doing most of the platform-invariant enforcement already, and further runtime-assertion work should focus on the narrow gaps (like opt-in migrations) that tests can't cover. **Going forward**: before proposing new enforcement, grep for existing coverage. "We already have a test for that" is the best possible audit finding.
  4. **Grace periods have to be designed for.** A fix that changes default behavior (user scope → project scope) cannot instantly invalidate every prior install. Users upgrade SVP by pulling git and re-running `sync_workspace.sh`, not by re-running the install wizard. Any fix that depends on a user-side migration must assume the migration has NOT been performed on the majority of upgrade paths and plan accordingly. S3-123 did the first half of this (the project-scope write is idempotent and safe to re-run). S3-128 does the second half (the user-scope leak is detected and reported so the user knows to finish the migration). **Going forward**: when a bug fix changes default behavior across an upgrade boundary, draft both halves of the migration in the same commit cycle — the new default AND the detection of systems still running the old default.
- **Pattern.** **P31 (NEW — Opt-In Migration Without Residual-State Detection)**. A bug-fix migration path is documented for the user to perform manually. The fix's own code runs correctly on freshly-migrated systems but has no runtime signal for systems that upgraded without performing the migration. Sibling of P30 (Invariant Documented But Not Enforced): both are "documentation carries the rest" failures, but P30 is about invariants the code should check on every run, while P31 is about state the code should check once per upgrade. Mitigation is the same shape: convert the documented step into a runtime detection function.
- **Cross-reference.** Output of the post-S3-127 runtime enforcement sweep. The sweep was triggered by the S3-127 lessons-learned cross-reference that said *"SVP's platform-contract layer is under-enforced. The next audit candidate is a comprehensive sweep of every assumption SVP makes about Claude Code's filesystem layout, cache structure, plugin discovery, and settings file precedence — encoded as runtime assertions rather than review-time documentation."* The sweep's scope shrank from four items to one as the dogfood checks and existing-coverage audits ran. **I11 (deployed artifact staleness) is deferred to a standalone bug entry S3-129, pending a Unit 23 assembly-flow audit.**

## S3-129: Deployed Artifact Round-Trip Regeneration Test

- **Bug (test-coverage gap).** Bug S3-80 (2026-04-04) added `regenerate_deployed_artifacts()` to Unit 23 and the `TestDeployedArtifactFreshness` class to `tests/regressions/test_bug_fix_completeness.py` with four byte-equal freshness tests. This closed the visible staleness categories (commands, orchestration skill, agent bodies, `hooks.json`) but left three gaps uncovered: (1) the four **hook shell scripts** (`write_authorization.sh`, `non_svp_protection.sh`, `stub_sentinel_check.sh`, `monitoring_reminder.sh`) had no byte-equal check against their `generate_*_sh()` source functions in Unit 17, (2) the **executable bit** (`chmod 0o755`) on the hook shell scripts was never verified — a stripped mode bit would make `non_svp_protection.sh` non-functional, which would unblock every Bash command in SVP sessions (the exact failure mode S3-80 was designed to prevent), and (3) **agent frontmatter drift on fields other than `name`**: `test_agents_match_source_pass2` used `source_content in deployed_text` (substring, not byte-equal), which asserts the body is present but is blind to drift on `description` or `model`. A deployed agent whose `description` was computed from a stale first-non-comment line, or whose `model` was pinned to a value `project_profile.json::agent_models` no longer referenced, would pass the existing test while being wrong.
- **Why the gap existed.** S3-80 wrote its test suite when the three most-visible staleness categories were commands, agent bodies, and `hooks.json`. Those had been the source of observed user-facing breakage. The hook shell scripts and agent frontmatter fields were less visible because (a) hook breakage manifests as permissive failure — bash commands run when they shouldn't, silently — and (b) agent `description` and `model` drift is tolerated by Claude Code's loader with no user-visible error. The tests covered what had been seen; they didn't cover what *could* be seen in the future. **Lesson: test coverage derived from a "tests the bug we just fixed" mindset tends to leave invariants uncovered in regions adjacent to the observed failure. Every bug-fix commit that adds regression tests should ask: what other invariants could drift in the same direction, and does my new test catch those too? If not, write the broader test now while the context is fresh.**
- **How this bug was found.** It was NOT found by a user-visible failure. It was found by the **post-S3-128 audit** — the conservative tail of the runtime enforcement sweep that produced S3-128. After S3-128 shipped, I11 (deployed artifact staleness) was the last candidate in the audit queue. The original plan for I11 was a runtime enforcement function in Unit 29's preflight, which would have been a large refactor with real blast radius. Discovery read of Unit 23's `regenerate_deployed_artifacts` revealed that the function was already self-contained and idempotent — and that `TestDeployedArtifactFreshness` already had four freshness tests covering the visible categories. The gap was smaller than expected, but real.
- **Dogfood pre-check saved effort and confirmed cleanliness.** Per the conservative protocol (originally formalized for S3-128), the round-trip test was run as a standalone script against the real `svp2.2-pass2-repo` BEFORE being committed to the test suite. The pre-check compared 87 deployed files (11 commands, 21 agents, 1 orchestration skill, 5 hooks including `hooks.json` + 4 shell scripts, plus a handful of regression-test files inside `svp/tests`) across content and mode and found **zero drift**. This confirmed (a) the round-trip design works correctly on real inputs, (b) the current repo is in the exact state `regenerate_deployed_artifacts` would produce, and (c) the test would land green rather than fire on historical drift. If the dogfood check had found drift, the fix cycle would have split into two — first fix the pre-existing drift, then land the test. No drift meant one-step landing.
- **Fix (Bug S3-129).** Single change: added `test_no_drift_via_roundtrip_regeneration` to `TestDeployedArtifactFreshness` in `tests/regressions/test_bug_fix_completeness.py`. The test copies `PASS2_REPO` to a `tmp_path` directory via `shutil.copytree(..., symlinks=False)`, calls `regenerate_deployed_artifacts(target)` on the copy, and walks both `svp/{commands,agents,hooks,skills}` trees asserting for every regular file (a) it exists in both trees, (b) the byte content matches, (c) `stat().st_mode & 0o777` matches on non-Windows platforms. `.DS_Store` and `__pycache__` are filtered from both sides. The test completes in ~2 seconds. **No production code changes. No refactor of `regenerate_deployed_artifacts`. No new helpers. No blueprint contract changes.**
- **Why round-trip rather than three new per-category tests.** The initial plan was to add three new tests: one byte-equal check for hook shell content, one mode-bit check for hook executability, one strengthened byte-equal check for agent full-file content including frontmatter. Each would need to re-derive the expected content using either duplicated assembly logic (a drift risk between test and production) or refactored helper functions extracted from `regenerate_deployed_artifacts` (a production code refactor, which introduces its own risk and must be behavior-preserving). The round-trip approach uses the production function itself as the oracle — the test contains zero duplicated frontmatter/content logic, zero knowledge of how `regenerate_deployed_artifacts` computes anything, and zero coupling to Unit 17 or Unit 18-22-24 signatures. It covers all three gaps with one assertion site, and automatically extends to any new assembly invariant the function ever gains.
- **Lessons.**
  1. **Round-trip regeneration is a strictly stronger test than per-category byte-equal checks.** Per-category tests verify "this specific output matches this specific source constant" — they lock the current set of invariants but are blind to new invariants the production function may gain later. A round-trip test verifies "the function is idempotent on the delivered state" — which is a universal quantifier over every invariant the function enforces, current and future. The round-trip test is *fewer lines of code* AND *covers more invariants* AND *requires less maintenance when the production function evolves*. When a production function is cheap to re-run and produces derived-state artifacts on disk, round-trip regeneration should be the default test shape. **Mitigation protocol**: whenever you are tempted to write a test that re-derives expected content from source constants, check whether the production function can be called on a temp-copy of the real state and compared to the original. If it can, do that instead.
  2. **"Tests what we just fixed" leaves adjacent invariants uncovered.** Bug-fix regression tests are usually narrowly scoped to the failure mode that was observed. This is efficient in the moment but creates a coverage pattern shaped like a leopard — spots of tight enforcement surrounded by regions of darkness. S3-80 had spots around commands and agent bodies; the shell scripts and agent frontmatter were in the dark regions. S3-129 is one round-trip test that illuminates the dark regions for the entire derived-artifact set at once. **Going forward**: every bug-fix commit that adds a regression test should include a follow-up question — *what other invariants could drift in the same direction as the one I just fixed?* If the question is nontrivial, the answer belongs in the same commit or a linked follow-up.
  3. **Dogfood pre-check is a one-step protocol, not an optional step.** Running the planned test as a standalone script against the real repo BEFORE committing costs 2 minutes and prevents an entire class of "shipped test fires on live data" bugs. The S3-128 protocol made this formal. S3-129 applied it correctly and confirmed cleanliness in one run. The dogfood check is not just for catching inverted rules (the S3-128 near-miss) — it's also for confirming that the design choice (round-trip in this case) works on real inputs before any other code changes. **The dogfood check is the first step of EXECUTE, not a test-authoring afterthought.**
  4. **Audits usually retire more work than they add — part 2.** The post-S3-127 sweep started with four MUST-tier items. Three were retired as already-covered (I1 by S3-121 tests, I2+I8 by S3-122 tests, I5 by S3-127 `_find_marketplace_root`). One shipped as S3-128. I11 shipped as S3-129 but shrank from "new runtime enforcement function in Unit 29 with preflight wire-in" to "one test in an existing class with zero production code changes." The audit's real output is this narrowing — the sweep *retired* ~90% of its initial scope. **Going forward**: when auditing for "invariants that could be runtime-enforced", weight "is there already a test for this" higher than "could we add a runtime check for this." The answer to the first question is usually yes for well-tested codebases, and when it is, the work becomes zero. When it isn't, the work is usually a test addition (smaller, safer) rather than a production helper (larger, riskier).
  5. **One test can absorb an entire SHOULD-tier item.** The original I9 item (hook shell script executable bits) was in the SHOULD tier and would have needed its own dedicated test. The round-trip approach absorbs it as a single line (`stat().st_mode & 0o777` comparison) within a loop that also handles content. This is a general pattern: when a test iterates over a set of artifacts, adding one more invariant to the per-item assertion costs one line, not one test. Prefer strong per-item assertions in iterated tests over separate single-purpose tests for adjacent invariants.
- **Pattern.** **P32 (NEW — Round-Trip Regeneration As Test Oracle)**. When a production function produces derived artifacts on disk and a test needs to verify those artifacts are not stale, the test can use the function itself as the oracle by (a) copying the state to a temp directory, (b) running the function on the copy, (c) comparing the copy to the original. This eliminates duplicated logic between test and production, automatically covers any invariant the production function enforces now or in the future, and keeps the test passive — it does not need to know how the function computes anything. Sibling of the "fixture realism" lesson from S3-127: both are about making test infrastructure mirror production infrastructure rather than paraphrase it. The round-trip approach is stronger because it requires zero paraphrase — the test just calls the thing and compares.
- **Cross-reference.** Final item in the post-S3-127 runtime enforcement sweep. The sweep started as four MUST-tier candidates (I1 command filename check, I2+I8 agent frontmatter check, I6 user-scope leak detection, I11 deployed artifact staleness) and concluded with two shipped bugs (S3-128 for I6, S3-129 for a narrowed I11). The other candidates — I1, I2+I8, I5, I7, I9 — were all retired as already-covered or absorbed. **The sweep's real output is the retirement table**: of the four initial MUST-tier candidates, one shipped as originally scoped (I6 → S3-128), one shipped drastically narrowed (I11 → S3-129, one test instead of a preflight-integrated helper), and two were retired entirely on discovering existing coverage. Of the three SHOULD-tier candidates, all three were retired (I5, I7) or absorbed (I9 → S3-129 mode-bit assertion). **Lesson recap**: proactive audits of documented-but-unenforced invariants, when performed with a "grep for existing coverage first" bias, usually discover that most invariants are already enforced somewhere in the regression test corpus — and the audit's value is the small residual it surfaces rather than the bulk of items it initially proposes.
