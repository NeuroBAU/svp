# Project Context: svp2.1

## Project Name

svp2.1 (Stratified Verification Pipeline, version 2.1)

## Domain

Software development tooling. A deterministically orchestrated, sequentially gated development pipeline where a domain expert authors software requirements in natural language and LLM agents generate, verify, and deliver a working Python project. The pipeline runs inside Claude Code as a plugin.

## Problem Statement

SVP 2.0 verifies that code works. It does not verify that code is well-formed. A project built by SVP 2.0 passes all tests, installs correctly, and functions as specified -- but the code may have inconsistent formatting, unused imports, style violations, missing type annotations, import ordering inconsistencies, and overly complex functions. These are not bugs; the code works. But they represent manual cleanup work that the pipeline should have handled automatically.

SVP 2.1 closes this quality gap by introducing pipeline-integrated quality gates: deterministic checkpoints that run formatting, linting, and type checking at defined points in the verification cycle. Quality tools auto-fix mechanically resolvable issues; residuals that tools cannot auto-fix are escalated to the producing agent for one re-pass. Every project built by SVP 2.1 is automatically formatted, linted, and type-checked during the build. Quality is a pipeline guarantee, not an opt-out feature.

SVP 2.1 also introduces:
- Delivered quality configuration (the human's quality tool preferences for their delivered project)
- Changelog support
- A restructured post-delivery debug workflow with delivered repo path tracking, lessons learned integration, and gated commit/push

SVP 2.1 is the terminal release of the SVP product line. The pipeline architecture is complete. Future development takes the form of language-directed variants (SVP-R, SVP-elisp, SVP-bash), each built by SVP 2.1 as a Python project.

## Intended Users

- **End users of SVP:** Domain experts who cannot write code, who use the pipeline to build Python projects from natural language requirements.
- **Builders of SVP 2.1:** The human-LLM pair using SVP 2.0 to construct this release.

## Key Architecture Concepts

### Quality Gates (New in 2.1)

Three mandatory, data-driven quality gates run at fixed positions in the verification cycle:

| Gate | Position | Operations |
|------|----------|------------|
| Gate A | Post-test generation, pre-red-run | format + light lint (E, F, I rules). No type check on tests. |
| Gate B | Post-implementation, pre-green-run | format + heavy lint + mypy (--ignore-missing-imports, unit level) |
| Gate C | Stage 5 assembly | format check + full lint + full mypy (cross-unit, no --ignore-missing-imports) |

- Gate composition is data-driven from `toolchain.json` (`gate_a`, `gate_b`, `gate_c` lists).
- All gates are mandatory. No opt-out.
- Auto-fix runs first; residuals trigger one agent re-pass; then fix ladder.
- Quality gate retry budget is separate from fix ladder retry budget.
- Quality tools write files via subprocess, bypassing hooks (dual write-path -- correct by design).

### Pipeline / Delivery Split

- **Pipeline toolchain:** Fixed. The tools SVP uses internally to build and verify.
- **Delivery toolchain:** Configurable via `project_profile.json`. The tools the delivered project presents to its end user.
- In a self-build (SVP building SVP), pipeline and delivery tools coincide but are generated by different mechanisms.

### Profile and Toolchain

- **`project_profile.json`:** Human-approved configuration capturing delivery preferences including a new `quality` section (linter, formatter, type_checker, import_sorter, line_length) and a `vcs.changelog` field.
- **`toolchain.json`:** Pipeline-internal configuration mapping operations to command templates. Gains a `quality` section with gate composition lists and quality package names.
- Profile is immutable after Gate 0.3. Changes require `/svp:redo`.

### Post-Delivery Debug Workflow (Restructured in 2.1)

Seven-step logic bug path:
1. Prompt received
2. Investigate in delivered repo
3. Fix workspace, then reassemble
4. Evaluate spec / blueprint
5. Regression test
6. Update lessons learned document
7. Commit and push (gated at Gate 6.5)

- `delivered_repo_path` recorded in `pipeline_state.json` at Stage 5 completion.
- Triage agent receives delivered repo path in task prompt.
- Delivered repo is writable after Gate 6.0.
- Debug commits use a fixed format regardless of `vcs.commit_style`.
- Lessons learned document (Doc 4) is a living document updated during debug sessions.

## Data Characteristics

SVP processes text artifacts:
- Stakeholder specs (Markdown)
- Blueprints (Markdown with embedded Python)
- Configuration files (JSON)
- Quality tool configuration (`ruff.toml`)
- Conversation ledgers (JSONL)
- Python source code and test suites

## Key Constraints

1. All SVP 2.0 pipeline behavior must be preserved: same stages, gates, fix ladders, verification cycle, pipeline/delivery split, profile mechanism, and toolchain abstraction.
2. Behavioral equivalence: resolved commands in existing toolchain sections must match SVP 1.2 / 2.0 hardcoded commands. The quality section is purely additive.
3. Quality gates are not new stages and do not add new agent types or gate types.
4. Unit numbering and dependency DAG preserved from SVP 2.0.
5. Hook enforcement uses hard blocking (PreToolUse exit code 2). Hook configuration schema validated against Claude Code's actual requirements.
6. Routing script action blocks emit explicit PREPARE and POST commands -- no improvisation by the main session.
7. Gate prepare commands use `--gate` flag with full gate ID; all gate IDs registered in `ALL_GATE_IDS` with handlers.
8. File copy operations guard against same-file collision when output path matches internal write path.
9. **Two-branch routing invariant (spec Section 3.6).** Every sub-stage where an agent completion triggers a gate presentation must have a two-branch check in `route()`: check `last_status.txt` to distinguish "agent not yet done" (invoke agent) from "agent done" (present gate). Applies universally to Stage 0 (project_context, project_profile), Stage 1 (stakeholder_spec_authoring), Stage 2 (blueprint_dialog, alignment_check), Stage 4 (integration test author), Stage 5 (git repo agent), redo profile sub-stages (redo_profile_delivery, redo_profile_blueprint), and all post-delivery debug loop agent-to-gate transitions (triage to Gate 6.2/6.4, repair to Gate 6.3, test agent to Gate 6.1). This is a structural invariant, not a per-stage fix. Must be applied universally in a single implementation pass, not incrementally (Bug 43 fix).
10. **Profile canonical naming invariant (spec Section 6.4).** The `project_profile.json` schema defines exact canonical section and field names. `DEFAULT_PROFILE` in the config reader, the setup agent's output, and all downstream consumers must use identical names. Mismatches (e.g., `licensing` vs `license`, `readme.target_audience` vs `readme.audience`) cause silent merge conflicts in `_deep_merge`.
11. **Launcher CLI vocabulary is fixed at three modes (spec Section 6.1.1).** `svp new <name>`, bare `svp` (auto-detect resume), and `svp restore <name> --spec --blueprint-dir --context --scripts-source --profile`. `svp restore` requires `--profile <path>` pointing to a valid `project_profile.json`. No other subcommands.
12. **Setup agent UX: four behavioral rules (spec Section 6.4).** (1) Plain-language explanations for every choice -- no jargon without definition. (2) Best-option recommendation for every choice -- clearly marked. (3) Sensible defaults that produce a correct project. (4) Progressive disclosure -- lead with recommendation, details only on request.
13. **Route-level state persistence invariant (spec Section 3.6).** Any `route()` branch that performs an in-memory state transition and then recursively routes must persist state to disk before returning the action block. Without this, `update_state.py` loads stale state from disk and overwrites the transition.
14. **Exhaustive dispatch_agent_status invariant (spec Section 3.6).** Every main-pipeline agent type in `dispatch_agent_status` must explicitly advance the pipeline state. A bare `return state` is only valid for slash-command-initiated agents (help, hint) whose completion does not require a stage/sub-stage change. The routing function and dispatch function must have consistent null-handling for `sub_stage` (Bug 44 fix).
15. **Exhaustive dispatch_command_status invariant (spec Section 3.6).** Every `dispatch_command_status` handler for `test_execution` must produce a state transition for the expected outcome at each sub-stage. No-op returns are invalid for `red_run` (TESTS_FAILED must advance to implementation) and `green_run` (TESTS_PASSED must advance to coverage_review) (Bug 45 fix).
16. **COMMAND/POST separation invariant (spec Section 3.6).** COMMAND fields in routing action blocks must never embed state update calls (`update_state.py`). State updates are exclusively the responsibility of POST commands. Embedding state updates in both COMMAND and POST causes double dispatch and TransitionError (Bug 47 fix).

## Success Criteria

1. All SVP 2.0 regression tests pass (updated for signature changes; covering same bug scenarios). All 35 regression test files carry forward (15 carry-forward from SVP 2.0, 7 carry-forward from SVP 2.1 prior builds, and 13 newly authored in this build).
2. Quality Gate A: post-test, pre-red-run. Format + light lint (E, F, I). No type check on tests.
3. Quality Gate B: post-implementation, pre-green-run. Format + heavy lint + mypy (--ignore-missing-imports, unit level).
4. Quality Gate C: Stage 5 assembly. Format check + full lint + full mypy (cross-unit, no --ignore-missing-imports).
5. All quality gates mandatory. Auto-fix first, 1 agent re-pass on residuals, then fix ladder.
6. Quality gate retry budget separate from fix ladder retry budget.
7. Gate composition data-driven from `toolchain.json`.
8. Quality tools bypass hooks (dual write-path, correct by design).
9. Setup agent dialog gains Area 5 (delivered quality preferences) and changelog question in Area 1.
10. Profile schema gains `quality` section and `vcs.changelog` field.
11. `ruff.toml` copied to project root by launcher; set to read-only immediately; permanently read-only thereafter.
12. Quality packages (`ruff`, `mypy`) installed during Pre-Stage-3 from `toolchain.json`.
13. Git repo agent generates delivered quality tool configuration from profile `quality` section.
14. Git repo agent generates changelog from `vcs.changelog` profile field.
15. `delivered_repo_path` recorded in `pipeline_state.json` at Stage 5 completion.
16. Post-delivery debug loop: 7-step workflow with delivered repo path in triage task prompt, writable after Gate 6.0.
17. Debug commits use fixed format regardless of `vcs.commit_style`.
18. Lessons learned document updated during debug sessions.
19. All new cross-unit paths covered by Stage 4 integration tests.
20. Two-branch routing invariant applied universally to every agent-to-gate sub-stage transition (not just Stage 0).
21. Profile canonical naming invariant: `DEFAULT_PROFILE`, setup agent output, and all consumers use identical section/field names from the canonical schema.
22. Launcher fully specified: ordered pre-flight checks with specific error messages, three CLI modes (`svp new`, bare `svp`, `svp restore`), session launch via `subprocess.run` with `cwd`/`env`/restart-signal loop.
23. Setup agent UX: four behavioral rules (plain-language explanations, best-option recommendations, sensible defaults, progressive disclosure) applied to all five dialog areas.
24. All carry-forward regression tests pass (34 files total: 15 carry-forward from SVP 2.0, 7 carry-forward from SVP 2.1 prior builds, and 12 newly authored in this build).
25. Route-level state persistence invariant: every `route()` branch that performs an in-memory state transition and recursively routes must persist state to disk before returning the action block. Exhaustive `dispatch_agent_status` coverage: every main-pipeline agent type must have an explicit state-advancing handler (not a bare `return state`).
26. Redo-triggered re-delivery must not collide with existing delivered repo directory (repo collision avoidance mechanism, spec Section 12.1).
27. Universal two-branch routing compliance: every entry in the Section 3.6 exhaustive list must have a corresponding two-branch check in `route()`, verified by a single structural regression test (`test_bug43_stage2_blueprint_routing.py`). Gate ID consistency: every gate ID in `GATE_VOCABULARY` must appear in `ALL_GATE_IDS` and vice versa.
28. Stage 3 dispatch completeness: `dispatch_agent_status` for `test_agent` handles `sub_stage=None` (Bug 44); `dispatch_command_status` for `test_execution` advances red_run->implementation and green_run->coverage_review (Bug 45); `dispatch_agent_status` for `coverage_review` advances to unit_completion (Bug 46); `unit_completion` COMMAND does not embed state update calls (Bug 47).
